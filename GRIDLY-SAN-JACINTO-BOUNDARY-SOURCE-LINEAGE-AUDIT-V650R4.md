**Bottom line:** the active boundary's exact coordinate source remains unproven. It was not generated by a direct exact extraction of the checked statewide `GEOID=48407` feature. The best-supported finding is **manual / synthetic / unknown provenance controlled-activation asset**.

## 5. Explanation of visual mismatch

The browser boundary is visually wrong because the file rendered by the active-only boundary path is not a proven authoritative San Jacinto County outline.

Specific causes found:

1. **Insufficient authoritative provenance.** The active file declares a Gridly V646.1 controlled-activation source, not a direct authoritative source export.
2. **No extraction command is tied to the committed active asset.** The repository contains a local extraction harness, but no report or committed command proves that the active file came from an authoritative `GEOID=48407` source.
3. **Statewide source mismatch.** The checked statewide `GEOID=48407` feature does not match the active file and is itself not credible.
4. **Likely synthetic or over-smoothed geometry.** The active polygon has 431 coordinates with visibly regular spacing and smoothed edges, consistent with a generated placeholder or simplified approximation rather than a production-grade county boundary.
5. **Not a CRS/order issue.** The active coordinates are valid longitude/latitude values in the expected east-Texas range, and the statewide feature's coordinates are also longitude/latitude values, but for a different incorrect location. The mismatch is source/geometry quality, not merely coordinate order.
6. **Not proven as wrong feature selection at runtime.** Runtime selection points to the San Jacinto active boundary file. The problem is the contents and provenance of that file, not evidence that Liberty or Montgomery is being rendered in its place.
