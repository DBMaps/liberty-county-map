# LP036.1 Passive Location Resolution Engine

LP036.1 adds a strictly passive, read-only resolver for supplied latitude/longitude pairs. It derives the county, awareness area, community, Houston parent/child-region context, and expected roadway runtime classification without applying the result.

## Resolver input contract

Call `window.gridlyResolveLocationContext?.({ latitude, longitude, source, requestedAt, correlationId })` with explicit coordinates. `latitude` and `longitude` are required. The resolver never requests browser geolocation, ZIP lookup, address search, or any other coordinate source.

## Resolution order

1. Normalize and validate coordinates.
2. Resolve supported county containment from the existing county awareness bounds contract.
3. Resolve awareness candidates from `GRIDLY_AWARENESS_AREA_DEFINITIONS` for that county.
4. Apply LP035 Houston regionalization only for Harris County.
5. Derive community identity from the matched awareness area.
6. Report expected roadway runtime context using the existing roadway runtime source resolver.
7. Snapshot current authoritative and persisted location owners read-only.
8. Compare the passive result with current state.
9. Record a bounded in-memory observation.

## Returned result contract

The resolver returns a normalized object with `available`, `status`, `input`, `coordinateValidation`, `support`, `county`, `community`, `houston`, `awarenessArea`, `roadwayRuntime`, `comparison`, and `mutationSafety` sections. Ordinary unresolved cases return structured statuses such as `invalid_coordinates`, `outside_supported_coverage`, `ambiguous_county`, `county_boundary_unavailable`, and `resolution_error` instead of mutating or silently defaulting.

## Existing authoritative owners

LP036.1 observes, but does not own or update, active county runtime state, selected awareness state, selected community state, settings, profile, legacy home-town storage, roadway runtime county, and Houston region preference state. Existing owners remain authoritative until a later migration phase.

## Houston behavior

Houston remains the Harris parent identity. LP035 child regions remain distinct awareness selections. Separate Harris communities such as Pasadena, Baytown, Humble, Katy, Cypress, Spring, Tomball, Deer Park, La Porte, Kingwood, Channelview, and Crosby are protected from accidental Houston-child classification.

## Runtime classification behavior

The engine calls the existing runtime-source resolver only to classify the expected runtime for the resolved county. It does not activate counties, load roadway datasets, request Harris partition packages, change active partitions, or touch cache state.

## Storage observation behavior

The resolver passively reads only location-relevant snapshots from `gridlySettingsV1`, `gridlyUserProfileV1`, `gridlyHomeTown`, selected county/community preference keys, and Houston region preference keys where available. It does not normalize, migrate, repair, or write any stored value.

## Passive safety guarantees

Mutation-safety counters are initialized to zero and reported with every resolution and audit. LP036.1 code does not call protected mutating owners for county activation, community selection, awareness selection, settings/profile/home-town writes, local/session storage writes, roadway loading, Harris partition loading, network refreshes, alert rebuilds, or map movement.

## Browser audit

Use `window.gridlyLp0361PassiveLocationResolutionAudit?.()` to inspect readiness, the current authoritative state snapshot, current storage-location snapshot, bounded observations, latest resolution fields, comparison mismatches, mutation counters, and LP036.2 readiness. Use `window.gridlyResetLp0361PassiveLocationResolutionAudit?.()` only to clear LP036.1 in-memory observations.

## Unresolved limitations

County containment currently uses the existing county awareness-boundary contract available in app memory. Awareness resolution uses existing center/radius and countywide awareness contracts where polygon containment is not yet available. Houston child-region containment is conservative and preserves separate-community exclusions; future phases should replace seed-radius matching with authoritative polygons when those contracts become available.

## Testing instructions

Run:

```bash
node --check js/app.js
node tests/lp036-1-passive-location-resolution.test.js
node tests/lp035-1-houston-regionalization-implementation.test.js
node tests/lp034r2-report-tap-map-correlation.test.js

```

Then perform browser validation with explicit coordinates only. Do not enable browser geolocation permission.

## Next-phase recommendations

LP036.2 should add migration/write-through behavior only after browser validation proves passive geographic correctness and all mutation counters remain zero. Priority follow-ups are authoritative polygon containment for counties and awareness areas, explicit storage migration planning, and consumer-facing confirmation UI that applies resolved state only by deliberate user action.
