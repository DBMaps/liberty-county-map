# LP036.1A County Containment Root-Cause Investigation

## 1. Executive Summary
LP036.1 is returning `ambiguous_county` for Dayton and Livingston because the resolver uses `GRIDLY_COUNTY_AWARENESS_BOUNDS_BY_ID`, an axis-aligned rectangular bounds contract, as automatic county containment truth. The resolver is not reversing coordinates, defaulting to an active/default county, or selecting a first/nearest candidate. It correctly reports ambiguity when more than one selectable operational county rectangle contains the coordinate.

**Conclusion B:** the current county bounds contract is insufficient and a dedicated county geometry milestone is required before LP036.2 can safely depend on coordinate-only county containment.

No production resolution behavior was changed in LP036.1A. The added browser helper is investigation-only and passive.

## 2. Browser Evidence
Added passive helper:

```js
window.gridlyLp0361aCountyContainmentAudit?.()
```

The helper reports `investigationOnly: true`, `productionBehaviorChanged: false`, matching counties and raw bounds for Dayton/Livingston/Houston controls, duplicate/malformed bounds metadata, polygon inventory status, and the root-cause classification.

## 3. Current County Containment Contract
`GRIDLY_COUNTY_AWARENESS_BOUNDS_BY_ID` is declared in `js/app.js` from individual county bounds constants and inline bounds objects. The object shape is `{ south, west, north, east, source }`, not Leaflet LatLngBounds array order and not `[west,south,east,north]` arrays. LP036.1 consumes it directly with inclusive numeric comparisons.

| Contract | Purpose | Geometry type | Authoritative for containment? | Current consumers | Known limitations |
|---|---|---:|---|---|---|
| `GRIDLY_COUNTY_AWARENESS_BOUNDS_BY_ID` | County awareness/map startup rectangular extents reused by LP036.1 county candidate containment | Axis-aligned bounding box, `south/west/north/east` | No | LP036.1 county resolver, map/display/audit code | Rectangles overlap across adjacent counties; cannot prove actual county ownership. |
| `GRIDLY_COUNTY_REGISTRY` | Runtime county metadata, names, operational/selectable flags, asset paths | Metadata plus map centers/bounds paths | No, except as ownership metadata | Runtime selection, package/source lookup, LP036.1 candidate filtering | Does not provide loaded polygon containment. |
| `GRIDLY_COUNTY_RUNTIME_SOURCE_REGISTRY` | Runtime source lookup for roads/crossings/boundaries | Source metadata | No | Runtime source bridge/resolvers | Source paths are not synchronous point-in-polygon containment. |
| County `boundaryPath` registry entries | County boundary asset inventory | GeoJSON file paths | Potentially, only after coverage/credibility/loading and point-in-polygon support are proven | Boundary overlays/package audits | Not currently loaded by LP036.1; point-in-polygon helper unavailable. |
| `GRIDLY_AWARENESS_AREA_DEFINITIONS` | Community and countywide awareness selection | Centers/radii and labels | No | Awareness/community resolution after county resolution | Awareness radii are consumer context, not county-line truth. |
| LP035 Houston regionalization | Houston parent/child community ownership | Region model/centers and ownership lists | No for county boundaries | Harris child-region assignment after Harris is resolved | Not applicable to county-line disambiguation. |
| Road/crossing package manifests | Runtime transportation sources | GeoJSON roads/crossings and manifests | No | Roadway/crossing runtime activation | Transportation features do not define county containment. |

## 7. County Bounds Data Quality Findings
The conflicting rectangles overlap in ordinary bounding-box space:

- Dayton overlap: Liberty × Chambers overlap is latitude `29.884282..30.13` and longitude `-95.11..-94.442235`. The Dayton test point is inside that overlap.
- Livingston overlap: San Jacinto × Polk overlap is latitude `30.489002..30.905` and longitude `-95.20018..-94.831`. The Livingston test point is inside that overlap.

The evidence points to expected rectangular bounding-box overlap, not duplicate IDs, malformed numeric fields, or coordinate-order corruption. Rectangles are fundamentally incapable of proving ownership near/within adjacent-county rectangular overlap because they include area outside true irregular county polygons.

## 8. Resolver Logic Trace
Coordinates → `gridlyLp0361ValidateCoordinates` validates finite latitude/longitude and ranges → `gridlyLp0361ResolveSupportedCounty` obtains selectable operational county IDs → `gridlyLp0361PointInBounds` tests each ID against `GRIDLY_COUNTY_AWARENESS_BOUNDS_BY_ID` using `south/west/north/east` → multiple candidates return `ambiguous_county` → only a single candidate proceeds to awareness/community/Houston/runtime classification.

Implementation review findings:

- No reversed latitude/longitude bug found.
- Inclusive edge comparisons are intentional but not the cause; Dayton/Livingston are clearly inside overlap, not edge-only cases.
- Filtering uses selectable/operational IDs before ambiguity classification.
- No default-county, active-county, first-match, or nearest-center fallback is present.
- No storage write, runtime activation, network fetch, or map movement is introduced.

## 9. Root Cause
Root cause classification: `inadequate_rectangular_geometry`.

LP036.1 is correctly exposing that its candidate source is not authoritative enough for county containment. The existing bounds contract is suitable as broad coverage/startup/awareness extent metadata, but not as the sole source of coordinate-only county truth.

## 10. Existing Authoritative Geometry Inventory
The repository contains registered county boundary GeoJSON paths for the 28 operational counties, but LP036.1 does not synchronously load them for passive containment, and no reusable browser point-in-polygon helper was found in `js/app.js`. Prior documentation also warns that some statewide/derived boundary files are not automatically trustworthy for county containment without credibility review. Therefore LP036.1A does not make a production polygon repair.

## 11. Repair Options
| Option | Assessment |
|---|---|
| A. Continue rectangles and preserve ambiguity | Geographically safe but leaves Dayton/Livingston unresolved; acceptable investigation default, not LP036.2-ready. |
| B. Rectangular narrowing + authoritative polygon containment | Best correctness path if complete credible polygons and point-in-polygon helper are packaged offline; moderate memory/startup cost; should be implemented in a dedicated geometry milestone unless existing contracts are proven runtime-ready. |
| C. Awareness-area inference | Not acceptable as county truth; centers/radii are consumer context and can overlap. |
| D. Community ownership inference | Not acceptable as county truth; community labels can span county lines and are not coordinate polygons. |
| E. ZIP-based resolution | Useful future adjunct for address/ZIP input, but not sufficient for coordinate-only GPS and not available passively here. |
| F. Dedicated lightweight county containment registry | Good future option if sourced from authoritative polygons and maintained consistently for all 28 counties; requires separate data milestone. |

## 12. Recommended Repair
Create a dedicated county geometry milestone before LP036.2. Preferred implementation: use current rectangular bounds only for candidate narrowing, then perform offline authoritative polygon containment against a complete, credible, packaged 28-county polygon registry or a derived lightweight containment registry. Do not use awareness/community inference as county truth.

## 13. Performance and Packaging Considerations
Polygon containment can be bounded by first using rectangles to reduce candidates. A lightweight simplified polygon registry may reduce browser memory and startup cost, but simplification must preserve county-line correctness and be documented with source/provenance.

## 14. Risks
- Continuing with rectangles can safely preserve ambiguity but blocks automatic county resolution for overlap areas.
- Using unvetted boundary files could create false positives with higher confidence than the current safe ambiguity.
- Inferring from communities, ZIPs, or nearest centers would guess and violate LP036.1 safety rules.

## 15. LP036.2 Readiness Decision
LP036.2 must remain blocked for automatic county containment until an authoritative, non-guessing county geometry contract exists and is tested across all 28 operational counties.

## 17. Rollback Strategy
Remove the investigation-only helper and `tests/lp036-1a-county-containment-investigation.test.js`. Because production resolver behavior was not changed, rollback does not require data migration or user-state cleanup.
