# LP036.1B — Authoritative County Geometry Foundation

## 1. Executive Summary

Implementation decision: **B — existing geometry assets are structurally valid and cover all 28 operational counties, but require a dedicated lightweight build/package step before production resolver integration.**

LP036.1B certifies that each operational county registry `boundaryPath` points to a local, parseable GeoJSON `FeatureCollection` containing one Polygon feature. The assets are suitable as source material for authoritative containment, and focused certification proves the Dayton and Livingston ambiguity cases resolve to Liberty and Polk respectively when polygon containment is applied. Production containment was **not** changed because the current runtime does not yet provide a dedicated offline PWA geometry package/cache contract.

## 2. LP036.1A Root Cause

LP036.1A proved `GRIDLY_COUNTY_AWARENESS_BOUNDS_BY_ID` is rectangular extent data. Dayton (`30.0466, -94.8852`) matches both Liberty and Chambers rectangles, and Livingston (`30.7110, -94.9327`) matches both Polk and San Jacinto rectangles. The resolver is correctly truthful when it reports rectangular ambiguity; the root cause remains `inadequate_rectangular_geometry`.

## 5. Geometry Format Certification

Certified at structural level: valid JSON, FeatureCollection top-level, one Polygon feature, closed rings through the source files, finite numeric longitude/latitude coordinates, coordinate ranges within world bounds, plausible southeast Texas bounding boxes, no duplicate registry owners, and county identity consistency by feature name vs registry name.

Interior rings are supported by the containment helper and explicitly tested with a synthetic hole fixture. Current operational source files did not require MultiPolygon certification as source data because they are all Polygon, but the helper supports and tests MultiPolygon.

## 6. Deployment Availability

The files are local repository assets. The Capacitor validation workflow creates `www/`, copies `index.html`, `manifest.json`, `service-worker.js`, `css`, `js`, `assets`, and `data`, then mirrors `www/` into `android/app/src/main/assets/public/`. Therefore the assets can be included in Capacitor packaging when that workflow is used.

However, `service-worker.js` precaches only the closure shell, manifest, icons, and logo. It does **not** precache county boundary assets. Therefore offline PWA availability is not yet proven for production resolver use.

## 7. Performance Analysis

| Option | Assessment |
| --- | --- |
| A. Load all 28 polygons at startup | Avoid: ~2.56 MB JSON plus parse cost on the main thread before first use. |
| B. Lazy-load one combined lightweight 28-county package | Preferred next architecture: one bounded request, one parse, versioned cache, easy PWA precache. |
| C. Rectangular narrowing then load candidate county assets | Good candidate-narrowing model, but raw per-county fetches can still add cache/request complexity. |
| D. Bundle simplified polygons in JavaScript | Offline-friendly but inflates JS and makes geometry diffs noisy. |
| E. Load/cache by region | Useful future scale strategy, but unnecessary complexity for 28 counties. |
| F. Use existing Leaflet boundary layers | Avoid: depends on rendered map state/DOM/layer lifecycle and can violate passive resolver isolation. |

## 8. Loading Strategy Options

Recommended: rectangular candidate narrowing followed by containment against a lazily loaded, versioned, lightweight 28-county geometry package. The parsed package should be cached in memory with a fixed upper bound and precached for PWA/Capacitor offline use. Failure mode must be `county_geometry_unavailable` or `county_geometry_parse_error`, never rectangle-only success.

## 9. Point-in-Polygon Contract

LP036.1B adds a small internal point-in-polygon helper for audit/certification. It supports Polygon, MultiPolygon, holes, deterministic boundary reporting, longitude/latitude coordinates, and no external library or network dependency.

## 10. Boundary-Edge Policy

- Strictly inside one county: resolve to that county.
- Inside a hole: no county match.
- On an outer or inner ring: return boundary.
- On a shared county border: future resolver should return `ambiguous_county_boundary` rather than guess.
- Multiple inside matches: future resolver should return `invalid_geometry_overlap` unless the match is exactly a shared boundary case.

## 11. Recommended Architecture

Create a dedicated `assets/location-resolution/gridly-authoritative-county-geometry-v1.json` generated from the certified boundary assets. It should contain only canonical county id, county name, geometry type, geometry, bounds, source path, source version, and certification status. Then precache it for PWA and include it in Capacitor `www` mirroring. Integrate LP036.1 only after that packaging contract passes browser/offline tests.

## 12. Production Implementation

Production containment was **not** integrated. LP036.1 passive resolver behavior remains unchanged and continues to report rectangular ambiguity truthfully.

## 13. Dayton Certification

Dayton (`30.0466, -94.8852`) has rectangular candidates `liberty-tx` and `chambers-tx`. Polygon containment against the certified source assets returns only `liberty-tx`.

## 14. Livingston Certification

Livingston (`30.7110, -94.9327`) has rectangular candidates `polk-tx` and `san-jacinto-tx`. Polygon containment against the certified source assets returns only `polk-tx`.

## 15. Full 28-County Certification

All 28 county assets parse and pass structural certification. Representative coordinate coverage is automated through required controls and all source assets are candidate-certified, but full production resolver certification remains blocked until the dedicated runtime package is created and the browser/offline validation is run against it.

## 16. Passive Safety Evidence

LP036.1B audit state reports zero state writes, zero storage writes, zero runtime activations, zero map movements, and zero network refreshes. The LP036.1B code does not fetch geometry or mutate application state.

## 17. Packaging/PWA/Capacitor Considerations

Capacitor packaging can receive assets via the existing CI copy step. PWA offline resolution is blocked because service worker precache does not include county geometry. A future geometry package milestone should add an explicit cache entry or another versioned offline asset strategy.

## 18. Known Limitations

- No production resolver integration in this milestone.
- No dedicated lightweight combined geometry package yet.
- PWA offline geometry availability is not proven.
- Existing source provenance/license metadata is limited in the files themselves; future package metadata should capture source and version explicitly.

## 19. LP036.2 Readiness Decision

`implementationReadyForLp0362` remains **false**. LP036.2 should not proceed until the dedicated geometry package is generated, offline packaging is validated, and the passive resolver is integrated with polygon containment and browser safety verification.

## 20. Rollback Strategy

The current milestone is additive and passive. Rollback is to remove the LP036.1B audit/helper code, the LP036.1B focused test, and this document. No user state, Settings, onboarding, map startup, active county, runtime, Supabase, reports, or roadway behavior was changed.
