# LP037.1 — Consumer Crossing Count Authority

## Scope

LP037.1 is an investigation and design milestone for the consumer meaning of `X crossings watched`. It does not change crossing counts, visibility, ownership, awareness boundaries, runtime loading, UI text, marker rendering, reporting, alerts, Route Watch, roadway runtime, county geometry, or Houston regionalization.

Spring Branch is used only as the certification example. The product rule below is intended to apply to every operational county, configured community, Houston child region, future Texas county, and future state expansion.

## Current ownership

| Stage | Current owner | What it owns today |
| --- | --- | --- |
| First owner | Certified county crossing package / normalized runtime inventory | Loads county crossing features, applies review visibility, and assigns the active county runtime inventory. |
| Runtime owner | `loadCrossings()` / `crossings` | Fetches active-county crossings, normalizes them, applies `shouldShowCrossingInLaunchMode()`, and sets `crossings` plus the bottom stat count. |
| Consumer owner | Awareness-area summary | Builds `crossingsInArea` from the selected awareness area, town/county anchor, or fallback behavior. |
| UI owner | Bottom panel / Awareness Brief copy model | Formats the consumer line as `X crossings watched` from the awareness summary display count. |
| Awareness Brief owner | Localized intelligence / active awareness model | May use a separate crossing-context count for Awareness Brief and mobile awareness metadata. |

## Observed divergence

LP037 found that certified inventory, runtime-selected crossings, consumer-policy crossings, and Awareness Brief counts can diverge. The Spring Branch example shows a runtime-selected set of 46 unique crossings while Harris has 1,159 inventory crossings and the overall certified production inventory contains 3,771 crossings with 0 duplicate FRA IDs.

The first divergence occurs after the certified crossing package becomes the runtime inventory: runtime inventory ownership and consumer awareness ownership separate when consumer surfaces derive counts from awareness-area geometry, town/radius matching, fallback nearest-crossing behavior, route-watch widening, viewport/render visibility, or Awareness Brief context instead of one explicit product authority.

## Candidate authorities

### Option A — Runtime-selected crossings

**Definition:** Count the active runtime crossing set returned by current selection logic.

**Pros**

- Simple to observe because the runtime already loads and normalizes this set.
- Aligns with current runtime-selected examples such as Spring Branch returning 46 unique crossings.
- Avoids requiring marker rendering to finish before a count can exist.

**Cons**

- Runtime selection can be a transport/loading implementation detail rather than a consumer promise.
- It may include fallback-selected or radius-selected crossings that do not match the consumer awareness geometry.
- It can diverge from marker visibility and Awareness Brief logic.

**Consumer implications:** Clear only if every surface explains that “watched” means runtime-selected, not visible or awareness-contained. Otherwise users can distrust mismatched map and brief counts.

**Future scalability:** Moderate. It is county-package friendly but brittle for communities, Houston child regions, and future nested regions unless runtime selection becomes explicitly region-aware.

### Option B — Consumer-visible crossings after production visibility policy

**Definition:** Count crossings that survive production reportability, public-roadway classification, awareness-area ownership, and visibility policy.

**Pros**

- Matches the crossings consumers are eligible to understand as public awareness coverage.
- Excludes non-reportable, hidden, private, rail-yard, and non-public industrial infrastructure when production policy marks them outside the consumer crossing surface.
- Provides one rule for county mode, community mode, Houston child regions, and future expansion.
- Separates “watched” from implementation details such as marker clustering, viewport hard caps, and load fallback.

**Cons**

- Requires a formal count API or shared selector so UI, Awareness Brief, alerts, and audits do not independently reimplement the filter.
- Migration must reconcile existing copy and bottom-panel fallbacks without changing counts prematurely.
- Needs fixtures for each region type to prevent future regressions.

**Consumer implications:** Strong. “Watched” means Gridly is applying its consumer crossing visibility policy to crossings owned by the user’s selected awareness area.

**Future scalability:** Strong. It works anywhere the product can define an awareness geometry and apply the same production visibility policy.

### Option C — Awareness Brief count owner

**Definition:** Count whatever the Awareness Brief model currently reports as crossing context.

**Pros**

- Optimized for the primary consumer narrative surface.
- Keeps the visible briefing internally coherent if only the brief is considered.
- Can incorporate richer context than a raw inventory count.

**Cons**

- The Awareness Brief is a presentation model, not the first owner of crossing inventory or policy.
- It can diverge from runtime, markers, and bottom-panel stats.
- It risks making a copy surface the source of truth for operational product semantics.

**Consumer implications:** Good within the brief, weaker across the map and other surfaces if counts differ.

**Future scalability:** Moderate to weak unless the brief delegates to a shared authority instead of owning the count.

### Option D — Existing authoritative owner: certified crossing inventory plus production consumer policy

**Definition:** Use certified package inventory as the data owner, but use production consumer visibility policy and awareness geometry as the product owner for the count.

**Pros**

- Does not invent a new data owner.
- Preserves certified inventory as the first owner while recognizing that consumer copy must be policy-filtered.
- Lets implementation create a single shared selector later without moving asset ownership.

**Cons**

- Existing code does not yet expose one shared authority function for every surface.
- Requires a repair sequence to consolidate runtime, consumer, UI, and Awareness Brief paths.

**Consumer implications:** Strong because it distinguishes inventory existence from consumer-watched coverage.

**Future scalability:** Strongest when formalized as the single product contract.

## Recommended authority

Recommend Option D as the product authority, implemented through the Option B counting rule: certified crossing inventory remains the first data owner, while the consumer-facing count is owned by the production consumer visibility policy applied inside the selected awareness geometry.

The authoritative phrase `X crossings watched` should mean:

> The number of certified, active-county crossing records that are reportable, consumer-visible under production crossing visibility policy, and geographically owned by the currently selected awareness area or county/community region.

## Recommended product definition

### What is counted?

Count unique crossing IDs from the certified active-county inventory after production launch visibility, reportability, public-roadway classification, and selected awareness geometry ownership all pass.

### What is excluded?

Exclude duplicate IDs, hidden review records, non-reportable grade-separated/non-highway records, records outside the active county inventory, and records outside the selected awareness geometry.

### When is a crossing watched?

A crossing is watched when Gridly has an active certified inventory record for it, the production consumer visibility policy allows it, and the selected county/community/Houston child region owns its coordinates for awareness purposes.

### Should private crossings count?

No, unless an existing production visibility policy explicitly classifies that crossing as consumer-reportable public-roadway coverage. Private-only infrastructure should not inflate a public consumer promise.

### Should industrial crossings count?

Only when the crossing is reportable, public-roadway-facing, and visible under production policy. Industrial-only or non-public operational trackage should be excluded.

### Should rail-yard crossings count?

No. Rail-yard crossings should be excluded unless they are separately represented as consumer-reportable public-roadway crossings by the production policy.

### Should temporary-access crossings count?

No by default. Temporary-access crossings should count only if certified inventory and production visibility policy mark them active, reportable, public-roadway-facing, and owned by the selected awareness geometry.

### Should crossings outside awareness geometry ever count?

No. Outside-geometry crossings may be loaded, cached, or used for operational diagnostics, but they should not count as watched for the selected consumer awareness area.

### Should fallback-selected crossings count?

No, not merely because fallback logic selected them. A fallback-selected crossing should count only if it also satisfies certified inventory, production visibility policy, and awareness geometry ownership.

## Spring Branch certification example

Spring Branch should not receive a special rule. If current runtime behavior returns 46 unique crossings, that number should be treated as a candidate runtime observation until the shared product authority confirms that the same 46 are certified, consumer-visible, and owned by Spring Branch awareness geometry.

The same discrepancy type can occur elsewhere because any county, community, or Houston child region can have separate runtime selection, awareness selection, marker visibility, and Awareness Brief context paths.

## Regional applicability

The recommended rule works for:

- all operational counties, because the county registry and certified crossing package remain first owners;
- all configured communities, because awareness geometry determines ownership before copy counts are shown;
- Houston child regions, because child-region geometry should own the watched count rather than the Harris parent inventory;
- county mode, because county-wide awareness geometry may own every policy-visible active-county crossing;
- community mode, because only policy-visible crossings inside the selected community awareness geometry count;
- future Texas counties and state expansion, because the rule depends on certified inventory, visibility policy, and selected awareness geometry rather than local hardcoded counts.

## Visibility policy relationship

The visibility policy is not a competing owner. It is the consumer eligibility gate applied after certified inventory ownership and before UI/Awareness Brief presentation. Marker rendering and viewport performance policy should not redefine the watched count.

## Migration considerations

1. Add a passive shared selector/audit that computes the recommended authority without changing UI text.
2. Add fixtures for county mode, community mode, Houston child regions, and known edge crossings such as private, industrial, rail-yard, temporary-access, and outside-geometry records.
3. Compare current runtime, consumer, UI, and Awareness Brief counts against the shared selector.
4. Repair copy surfaces to read the shared authority only after discrepancies are understood.
5. Only then adjust Spring Branch or any other production count.

## Future repair sequence

1. Freeze this definition as the product contract.
2. Inventory every code path that produces a crossing count.
3. Build a passive count-authority selector from certified inventory, production visibility policy, and awareness geometry.
4. Certify the selector against Spring Branch, all operational counties, all configured communities, and Houston child regions.
5. Migrate runtime consumer count, bottom-panel count, Awareness Brief count, reporting context, and audits to the shared selector.
6. Remove legacy fallback count ownership once certification passes.
