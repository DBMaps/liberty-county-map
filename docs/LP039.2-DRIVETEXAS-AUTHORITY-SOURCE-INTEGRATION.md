# LP039.2 — DriveTexas Authority Source Integration

LP039.2 connects the already-loaded production DriveTexas normalized records into the LP039.1 authority foundation. LP039.2 does not migrate consumer surfaces.

## Existing production source path

1. `gridlyDriveTexasLiveConnector` owns the DriveTexas endpoint template, timeout, retry count, fetch lifecycle, refresh cadence, retained source dataset, awareness-derived view, and refresh notification boundary.
2. `gridlyDriveTexasProvider.normalizeRecords()` owns raw GeoJSON/record extraction and normalized DriveTexas records.
3. `gridlyDriveTexasConnector.getAllNormalizedRecords()` exposes retained normalized source records; `getNormalizedRecords()` and `getAwarenessRecords()` expose the current awareness presentation view.
4. `gridlyDriveTexasConnector.areaLifecycleAudit()` exposes retained-record timestamps, last successful fetch, selected awareness derivation, retained fallback, and fetch failure evidence.
5. Official roadway/situation publication remains downstream presentation/enrichment and is not an authority source for LP039.2.

## Provider and connector owners

- Endpoint owner: `gridlyDriveTexasLiveConnector`.
- Fetch lifecycle owner: `gridlyDriveTexasLiveConnector.fetchNow()` / `startPolling()`.
- Normalization owner: `gridlyDriveTexasProvider.normalizeRecords()`.
- Retained-record owner: `gridlyDriveTexasConnector.getAllNormalizedRecords()`.
- Last-successful fallback owner: retained `allNormalizedRecords` plus `areaLifecycleAudit()` lifecycle evidence.

## Source-record resolver

`gridlyGetLoadedDriveTexasAuthoritySourceRecords()` is read-only and passive. It prefers retained normalized connector records, then connector awareness records, then provider normalized records. It returns source-state evidence including provider/connector availability, enabled states, fetch failure, last refresh, last successful refresh, last error, fallback use, fallback reason, no-loaded-record state, and record count.

## Source adapter contract

`gridlyAdaptDriveTexasRecordsForAuthority(records, options)` converts existing normalized DriveTexas records into the LP039.1 authority input contract. It preserves available provider, identity, category, prose, lifecycle, coordinate/geometry, roadway, locality, URL, source metadata, connector-retained, fallback, refresh timestamp, and awareness-filter evidence fields without fetching or rendering.

## Source field preservation and limitations

Fields already present at the integration boundary are copied into authority records. Missing upstream fields are reported in `sourceFieldsUnavailable`; derived authority fields are reported in `sourceFieldsDerived`; fallback-only paths are reported in `sourceFieldsFallbackOnly`. LP039.2 does not fabricate county, city, district, update time, geometry, lane impact, status, expiration, locality, or roadway ownership.

## Geographic ownership integration

Adapted records feed LP039.1 `gridlyEvaluateDriveTexasGeographicOwnership()`. Source coordinates and geometry are preserved as the strongest available evidence. County/community metadata, roadway association, radius, and text remain lower tiers. Fallback ownership is disclosed and is not silently promoted to authoritative consumer eligibility.

## Roadway ownership integration

Adapted records preserve route name, roadway, canonical road, direction, line geometry, and source roadway metadata where available. LP039.2 does not load or recompute roadway packages and does not change Harris roadway package selection or cache behavior.

## Freshness and lifecycle integration

Adapted records preserve source start/update/end/expiration fields and connector refresh evidence. LP039.1 freshness evaluation distinguishes active, expired, stale, future-effective, missing timestamp, provider unavailable, connector unavailable, fetch failed, no loaded records, and retained fallback conditions.

## Retained-record fallback handling

When retained last-successful connector records are used, `sourceFallbackUsed` and `sourceFallbackReason` disclose that condition. Retained data is not classified as current without record freshness evidence.

## Deduplication and identity

Authority identity prefers stable provider/source/global IDs, then event IDs, then a deterministic source fallback key. Duplicate counts, unique provider counts, identity method counts, and fallback identity counts are reported. Same-title records with distinct stable IDs remain distinct.

## Category integration

All seven normalized DriveTexas categories flow into authority counts and eligibility counts: Crash, Closure, Flooding, Construction, Lane Closure, Bridge Restriction, and Travel Advisory. The provider's `Road Closure` normalized category is reported as `Closure` for LP039.2 category-count compatibility without changing consumer wording.

## Consumer eligibility

LP039.2 evaluates eligibility with production source evidence while preserving the LP039.1 rule that fallback-owned records are not silently consumer eligible. No markers, alert rows, briefs, counts, summaries, or quiet-state copy are migrated.

## Authority snapshot

`gridlyGetDriveTexasAuthoritySnapshot()` now supports a no-argument read-only live snapshot from loaded DriveTexas source records. The snapshot includes source integration status, source owner, counts, category counts, observed ownership/freshness/identity methods, provider/connector state, quiet-state reason, and the canonical authority result.

## Passive update boundary

LP039.2 computes snapshots lazily on demand. It adds no timer, interval, event loop, listener chain, network request, storage write, map movement, rendering call, or DriveTexas activation.

## Official situation boundary

Official situations remain presentation-only. LP039.2 reports `officialSituationIntegrated: false`, `officialSituationAuthorityOwner: false`, and `officialSituationPresentationOnly: true`.

## Consumer migration boundary and quiet-state contract

LP039.2 does not change DriveTexas markers, popups, alerts, Awareness Brief, Community Pulse, Travel Brief, Know Before You Go, active-condition summaries, county/community/Houston summaries, visible counts, or consumer quiet-state copy. Quiet-state reasons are authority-snapshot evidence only.

## Browser audit

`window.gridlyLp0392DriveTexasAuthoritySourceIntegrationAudit?.()` is passive and reports architecture, source-state, field-preservation, category, ownership, freshness, deduplication, eligibility, and migration-boundary evidence.

## Deterministic tests

`tests/lp039-2-drivetexas-authority-source-integration.test.js` uses deterministic test-only fixtures for seven categories, duplicates, stale/expired/future/missing timestamps, fallback identity, point/line/county/roadway/text evidence, retained fallback, disabled provider/connector, fetch failure, and no-loaded-record behavior. Fixtures are not injected into production runtime.

## Known limitations

LP039.2 can only preserve fields already present on normalized records. Existing provider normalization may not expose full line geometry, lane-impact details, district, status, provider URL, or complete source metadata for every DriveTexas payload.

## LP039.3 readiness

LP039.3 can migrate consumer DriveTexas ownership to the authority result because LP039.2 now exposes production source records, source lifecycle state, ownership evidence, freshness, identity, category counts, and eligibility in a single passive snapshot while leaving consumer surfaces unchanged.

## No-production-consumer-change statement

LP039.2 does not migrate consumer surfaces and does not alter DriveTexas endpoint, provider activation, connector activation, fetch cadence, polling cadence, timeout, retry, refresh lifecycle, marker rendering, alert rendering, official situation rendering, roadway runtime, county geometry, Houston regionalization, weather, crossings, community reports, hazards, Route Watch, Supabase, onboarding, settings, service worker, or Capacitor behavior.

## LP039.2R1 Woodville authority eligibility repair

Live browser validation for Woodville (`tyler-tx`) found 698 normalized DriveTexas records, 633 outside-awareness records, and 65 authority-eligible records for a configured seven-mile awareness radius. LP039.2R1 treats that result as uncertified unless every eligible record has record-level proof for identity, freshness, and selected-awareness geographic ownership.

### Root cause

The LP039.2 adapter previously delegated selected-awareness ownership to the LP039.1 foundation summary and surfaced record-level method arrays directly. That allowed `exact_source_geometry` to mean only that a source coordinate or geometry existed; it did not prove the point was within the selected Woodville radius. The audit also returned repeated method/status arrays and did not expose `identityMethodCounts`, leaving the 65 Woodville records unexplained at record level.

### Coordinate contract

DriveTexas source points are accepted only when finite coordinates resolve inside Texas latitude/longitude ranges. GeoJSON coordinate arrays are interpreted as `[longitude, latitude]`; application arrays may be interpreted as `[latitude, longitude]` only when that order resolves to Texas. Coordinate objects use `{ latitude, longitude }`/`{ lat, lng }`. Reversed-coordinate suspects, malformed coordinates, and missing coordinates cannot establish authority eligibility.

### Distance contract

Selected-awareness radius is expressed in miles. LP039.2R1 calculates point distance geodesically with the haversine formula and compares the result to the configured awareness radius. Seven miles is never treated as degrees, kilometers, retained-record membership, route association, or text evidence.

### Accepted ownership and AND semantics

Authority eligibility now requires all of the following conditions with AND semantics: unique identity, consumer-meaningful DriveTexas category, valid Texas source point, valid selected-awareness anchor, point distance inside the configured awareness radius in miles, no reversed-coordinate suspect, and active freshness proof. Route name, county/city/headline/description text, provider membership, connector retention, category, map visibility, or freshness alone remains non-authoritative.

### Audit summary repair

`window.gridlyLp0392DriveTexasAuthoritySourceIntegrationAudit?.()` now exposes concise method/status inventories plus `identityMethodCounts`, ownership/freshness/eligibility/ineligibility count objects, record-level proof, eligible-record proof, distance bands, coordinate-quality counts, eligible-distance statistics, selected radius, and certification booleans. `window.gridlyLp0392DriveTexasEligibilityProofAudit?.()` is passive and returns current selected-awareness proof without fetching, polling, rendering, writing, moving the map, activating providers, or changing consumer output.

### Final live eligibility result and limitations

The repaired logic does not force Woodville's eligible count to zero; it accepts only records whose proof satisfies the AND contract. A live browser rerun is still required to capture the exact final Woodville count from the current DriveTexas feed because tests use deterministic fixtures and the repository does not include the live 698-record snapshot. Remaining limitations: LP039.2R1 certifies source points and existing foundation-supported explicit ownership only; unavailable line geometry, county, city, canonical road, status, expiration, and awareness evidence fields cannot be used as silent ownership proof.
