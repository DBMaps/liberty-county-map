# LP039.3R2 DriveTexas Roadway Impact Authority Investigation

## 1. Executive conclusion

LP039.3R2 is investigation-only. It does not repair production authority or make Dayton show DriveTexas records. The current authority contract admits consumer-visible DriveTexas records only when a fresh, unique, consumer-meaningful record has a valid source point inside the selected awareness radius.

The current live Dayton zero-result state is best classified as `mixed_root_cause` when loaded records include retained source extent, and as `provider_points_outside_awareness_and_no_extent_available` when the loaded normalized records contain only points. Current code evidence shows that the earliest in-repo DriveTexas provider accepts GeoJSON FeatureCollection payloads and temporarily attaches each feature geometry as `__geometry`, but provider normalization emits latitude/longitude, route, timestamps, and source trace while omitting the full geometry and raw limit fields. LP039.2 then evaluates authority by center-to-point distance only.

Recommendation: use Option D, a layered geographic authority contract, in a future milestone. Accepted future methods should remain independently geographic and auditable: trusted point containment, trusted source geometry intersection, and/or a certified affected-segment intersection chain. Do not widen radius, use route-name-only matching, promote provider prose, or add Dayton/Liberty/community-specific logic.

## 2. Current point-radius contract

Current LP039.2 eligibility is stricter than the LP039.1 foundation. LP039.2 builds per-record proof and sets `geographicOwnershipMethod` to `valid_source_point_inside_awareness_radius_miles` only when a valid source point is within the selected awareness radius. Records outside the radius receive `not_established`. Final eligibility also requires uniqueness, non-retained source evidence, consumer-meaningful category, valid coordinate order, selected awareness anchor, point containment, and active freshness.

Accepted today:

- Valid source point inside selected awareness radius.

Rejected today:

- Source line/polygon intersection.
- Start/end or limit text intersection.
- Certified roadway segment intersection.
- Awareness polygon/boundary intersection.
- County/community metadata alone.
- Route-name-only matching.
- Provider prose/headline matching.
- Proximity-only/corridor relevance.

The selected-awareness radius is measured from the center anchor. Roadway ownership enriches evidence and presentation but does not make a record eligible.

## 3. Raw source payload findings

The earliest available in-repository DriveTexas provider payload shape accepts:

- A GeoJSON FeatureCollection with `features`.
- Each feature's `properties` object.
- Each feature's `geometry`, retained temporarily as `__geometry`.
- Alternate arrays named `incidents` or `records`.

Observed raw/upstream fields accepted by the in-repo normalizer include:

- Source/provider ID candidates: `GLOBALID`, `globalId`, `id`, `event_id`, `eventId`.
- Category/type candidates: `condition`, `event_type`, `eventType`, `type`, `category`.
- Headline/title/description candidates: `description`, `summary`, `title`, `headline`.
- Route/roadway candidates: `route_name`, `routeName`, `roadway`, `road`, `highway`.
- Coordinates from GeoJSON `Point` coordinates, GeoJSON `LineString` midpoint, or lat/lon fields.
- Geometry object from `feature.geometry`, `record.__geometry`, `record.geometry`, or `record.rawGeometry`.
- Start/end lifecycle timestamps from `start_time`, `startTime`, `start`, `beginDate`, `end_time`, `endTime`, `end`, `endDate`.

Fields not emitted by the current provider normalized record include full line geometry, MultiLineString geometry, polygon geometry, bounding box, start/end coordinates, raw county/city/district, mile markers, project/event extent, closure limits, provider links, and complete raw properties unless another connector has already carried them forward.

## 4. Geometry preservation trace

Trace by stage:

1. Raw source: GeoJSON feature geometry can exist as `feature.geometry` for FeatureCollection payloads.
2. Connector/load state: normalized records are retained by the connector via provider output; connector APIs return normalized records, not raw payload dumps.
3. Provider normalization: point coordinates are emitted directly; LineString geometry is reduced to its midpoint coordinate; full `__geometry` is not emitted in the normalized provider record.
4. LP039.2 authority adaptation: geometry is preserved only if already present as `geometry`, `roadwayGeometry`, or `routeGeometry` on loaded records.
5. LP039.1 authority selection: foundation may label source geometry as available, but LP039.2 supersedes consumer eligibility with point-radius proof.
6. LP039.3 consumer projection: consumes LP039.2 eligible records and proof; it does not resurrect discarded geometry.

First evidence-loss stage for provider-owned GeoJSON extent: `gridlyDriveTexasProvider.normalizeRecord`, where `__geometry` is converted to a latitude/longitude point and omitted from normalized output. If a connector-retained normalized record still has geometry, the first authority loss is later: LP039.2 authority selection ignores non-point extent for eligibility.

## 5. Dayton nearest-record findings

The new passive audit returns a concise `daytonNearestRecords` array with at most 25 records. For each record it reports ID, category, headline, route, raw/normalized coordinate, distance to Dayton anchor, distance outside the Dayton radius boundary, raw geometry type, downstream geometry retention, start/end/limit fields, county/community fields, source point containment, whether trusted geometry intersection is proven, certified-roadway proof status, whether route-name-only proof was avoided, anchor/extent suspicion, evidence classification, and unresolved reason.

Current supplied live evidence says the nearest loaded record is approximately 12.213 miles from the Dayton anchor, Dayton radius is 8 miles, and there are zero records inside the selected-awareness point radius. That puts the nearest source point approximately 4.213 miles outside the point-radius boundary.

The new audit reports whether a selected awareness object exposes a polygon, boundary, or bounds, but those shapes are not used by current DriveTexas authority selection. The audit does not create production geometry and does not mutate application state.

For Dayton validation, the known selected awareness representation is:

- Current LP039.2 method: center point radius.

## 7. Certified roadway proof findings

Certified roadway geometry could safely contribute only if it proves the full chain:

1. DriveTexas event to affected certified roadway segment.
2. Affected certified roadway segment to selected awareness geometry.
3. Both links without relying on route-name-only equality.

Safe evidence includes event source point lying on/near a certified segment, trusted source line intersecting certified geometry, retained start/end coordinates mapping to certified geometry, or affected segment geometry overlapping the selected awareness.

Unsafe evidence includes route-name equality alone, headline/prose, nearest roadway without event extent, corridor popularity, commuter assumptions, or subjective local knowledge.

The passive audit reports whether certified roadway runtime globals appear available, but it does not perform production linking or mutate roadway runtime loading.

## 8. Root-cause classification

Classification: `mixed_root_cause` when source line/extent evidence is present but not retained/used; otherwise `provider_points_outside_awareness_and_no_extent_available` for the loaded normalized point-only dataset.

Supporting evidence:

- Current consumer eligibility is point-radius only.
- Known Dayton live records have zero point containment.
- Route names near Dayton do not prove authority.
- Raw GeoJSON geometry may exist upstream, but provider normalization can flatten LineString to midpoint and omit full geometry.
- Certified roadway proof is not an accepted eligibility gate today.

Confidence: medium with loaded records; lower when raw live payload is unavailable in browser memory.

Unresolved questions:

- Which exact live DriveTexas records currently include trusted LineString, MultiLineString, Polygon, start/end coordinates, or usable limit extents upstream.
- Whether Gridly's certified roadway dataset can prove event-to-road ownership spatially without route-name-only dependency.

## 9. Authority option comparison

### Option A — Retain strict point containment

- Correctness: safest when source point is the authoritative event location.
- False-negative risk: high for roadway events where provider points are anchors/midpoints outside awareness but affected segments cross the awareness area.
- False-positive risk: low.
- Data requirements: minimal.
- Scalability: high.
- Houston child-region suitability: clear but potentially over-excludes segment impacts.
- LP039.2 compatibility: fully compatible.

### Option B — Source geometry intersection

- Correctness: strong if trusted source geometry is retained and represents affected extent.
- False-negative risk: lower than point-only.
- False-positive risk: controlled if geometry semantics are verified.
- Data requirements: retained trusted source geometry and awareness geometry or accepted radius/boundary model.
- Scalability: good with capped geometry operations.
- Houston child-region suitability: strong if child-region geometry is available.
- LP039.2 compatibility: requires future production change.

### Option C — Certified roadway impact chain

- Correctness: strong only when both event-to-road and road-to-awareness are spatially proven.
- False-negative risk: medium if event-to-road linking is incomplete.
- False-positive risk: high if implemented as route-name-only matching; low if fully spatial.
- Data requirements: certified roadway geometry, event geometry or source point proximity, and selected-awareness geometry.
- Scalability: feasible with indexed/capped spatial operations.
- Houston child-region suitability: potentially excellent.
- LP039.2 compatibility: requires future production change.

### Option D — Layered geographic authority

- Correctness: best future direction because independent methods can each remain explainable.
- False-negative risk: lower than point-only.
- False-positive risk: controlled if every layer is geographic and auditable.
- Data requirements: varies by accepted layer.
- Scalability: good with explicit caps and method-specific proof.
- Houston child-region suitability: best of the evaluated options.
- LP039.2 compatibility: requires a future authority-contract milestone.

### Option E — Radius expansion or corridor relevance

Rejected. Arbitrary widening, commuter assumptions, and corridor popularity are not selected-awareness ownership proof. Route relevance may inform route intelligence, but it must not establish consumer-visible DriveTexas authority.

## 10. Recommended future authority contract

Recommended direction: Option D. Build a staged layered contract that can accept:

- Trusted source point containment.
- Trusted source geometry/affected segment intersection.
- Certified affected-segment intersection only when the event-to-road and road-to-awareness links are both spatially proven.

Minimum safe production change, if any: a future milestone should first retain trusted source geometry and extent evidence through normalization and authority adaptation, then add auditable eligibility methods without changing consumer wording or quiet-state behavior by default.

## 11. Explicit non-recommendations

Do not:

- Widen the radius arbitrarily.
- Add Dayton-specific, Liberty-specific, county-specific, community-specific, or roadway-specific logic.
- Promote route-name-only matches.
- Promote provider prose/headline text.
- Restore raw provider, connector, official-situation, proximity-only, or legacy fallback paths.
- Use commuter assumptions or local knowledge.
- Change current consumer zero-result copy.

## 12. Merge implications for LP039.3

LP039.3 can eventually merge unchanged only if product accepts strict point-radius authority as the current safe contract and understands the false-negative risk. If roadway-impact authority is required, LP039.3 should not merge as the final DriveTexas authority state until a separate production repair is implemented and validated.

Default recommendation: do not merge LP039.3 or LP039.3R2 into main as completion of DriveTexas authority until the authority decision is explicitly accepted and any required repair is separately completed.
