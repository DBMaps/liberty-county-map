# LP053.2 — Historical Payload Minimization

## Problem

The Phase 1A passive historical capture path previously allowed complete community report snapshots to flow into `envelope.report`, `historical_events.payload`, and `historical_events.envelope`. Those snapshots can include raw device identifiers, exact coordinates, free-form detail text, synthetic hazard IDs that embed device identifiers, UI-only fields, and internal operational metadata that are not required for future aggregate historical intelligence.

## Capture Path Inventory

The current passive sidecar path is:

1. `gridlyTryPassiveHistoryCapturePhase1A()` in `js/app.js` runs after existing report success paths and fails open if the sidecar is unavailable.
2. `capturePhase1AEvent()` checks disabled-by-default flags, accepts only `report_created` and `report_cleared`, and passes the report-like object to the envelope builder.
3. `buildPhase1AEnvelope()` is the narrow canonical boundary where `sanitizePhase1AHistoricalReport()` minimizes the report before any envelope is returned.
4. `createPhase1AIdempotencyKey()` hashes only envelope fields that now contain the sanitized report representation.
5. `writePhase1AEnvelope()` validates the envelope and writes `payload: envelope.report` plus `envelope`, so both row columns receive only the minimized report object.

The sanitizer boundary is intentionally in the envelope builder because that point is before `envelope.report`, row `payload`, row `envelope`, and idempotency-key construction.

## Scope

LP053.2 changes only the passive historical capture sidecar, focused tests, and this documentation. It does not change public report inserts, public report schema, live Supabase data, report submission behavior, crossing behavior, hazard behavior, clear or confirmation flows, awareness filtering, alerts, integrations, PWA behavior, Route Watch, historical reads, historical UI, migrations, exports, or analytics.

## Approved Historical Field Allow-List

The sanitizer preserves only explicitly approved primitive fields and simple nested generalized-location output when present and valid:

- Classification and lifecycle: `reportType`, `reportState`, `lifecycle`, `category`, `subtype`.
- Source/evidence: `source`, `confidence`.
- Stable identifiers: safe `id`, safe `crossingId`.
- Crossing labels: `crossingName`, `railroad`.
- Hazard grouping labels: `roadName`, `intersection`, `countyId`, `state`, `communityId`.
- Time fields: normalized `sourceTimestamp` and `visibilityExpiresAt` only.
- Location grouping: `locationBucket` and, for non-crossing reports, `generalizedLocation`.

## Removed Sensitive or Unnecessary Fields

The sanitizer excludes unknown fields by default, including raw `device_id`, `deviceId`, raw device UUIDs, unsafe synthetic IDs, `detail`, free-form notes, full report copies, Supabase rows, internal metadata blobs, HTML, DOM or map objects, functions, raw latitude/longitude field names, user/session/localStorage identifiers, debug fields, audit-only fields, UI copy, original payload recursion, and envelope recursion.

## Location Minimization Policy

Historical payloads round report coordinates to 3 decimal places and expose them as a `locationBucket` string. Road-hazard payloads also include `generalizedLocation` with `{ precision: 3, lat, lng, bucket }`. Three decimal places are approximately 111 meters of latitude and roughly 95–100 meters of longitude in Gridly’s Southeast Texas operating region, which is sufficient for coarse roadway or intersection grouping while avoiding exact incident coordinates. This bucket is not an incident identity guarantee and must not be presented as exact location evidence.

For railroad crossings, certified crossing identity is preferred. When a safe crossing ID is available, exact crossing coordinates are omitted from the nested generalized-location object; a coarse `locationBucket` may still be retained only for broad grouping.

## Synthetic ID Policy

Hazard IDs matching device-derived client formats such as `hazard-...-<timestamp>` and `hazard-cleared-...-<timestamp>` are excluded from sanitized historical payloads. The detector does not parse, reverse, extract, or log the embedded identifier. Certified crossing IDs and ordinary stable source IDs remain usable unless they match those unsafe hazard patterns.

## Idempotency Policy

`createPhase1AIdempotencyKey()` hashes a canonical object built from schema version, event type, observed time, safe sanitized identifiers, report type, location bucket, and the sanitized report. It no longer needs to inspect raw report snapshots, raw device identifiers, free-form detail, or exact coordinates because `buildPhase1AEnvelope()` always places the sanitized report into the envelope first. Hash inputs are not logged.

## Fail-Open Behavior

Historical capture remains fail-open. Unsupported event types, disabled flags, malformed envelopes, missing writers, sanitizer exceptions, idempotency failures, and writer/storage failures return safe no-op or fail-open results and do not block normal report submission. Sanitizer failure returns a minimal empty report object rather than falling back to the raw report.

## Non-Goals

LP053.2 does not authorize historical reads, historical UI, exports, business analytics, incident identity redesign, live Supabase verification, schema changes, migrations, or changes to existing production records.

## Audit Helper

A passive browser-console audit helper is available:

```js
window.gridlyLp0532HistoricalPayloadMinimizationAudit?.()
```

It performs local in-memory checks only. It does not fetch, write, modify storage, modify the DOM, submit reports, call Supabase, or print sensitive sample values.

## Tests

Focused regression coverage lives in `tests/history-capture/historyCapturePayloadMinimization.test.js` and covers crossing sanitation, road-hazard sanitation, hazard-clear sanitation, input immutability, unknown-field exclusion, idempotency safety, fail-open behavior, flag preservation, and protected-system smoke checks. Existing Phase 1A envelope, idempotency, writer, and coordinator tests were also updated or run.

## Remaining Live Deployment Unknowns

This repository-side milestone does not claim live Supabase deployment verification. Prior tunnel 403 constraints mean live database schema exposure, grants, RLS, deployed artifacts, and existing rows remain unverified from this environment.
