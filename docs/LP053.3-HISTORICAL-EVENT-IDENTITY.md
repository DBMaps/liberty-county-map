# LP053.3 — Durable Historical Event Identity

## Purpose

LP053.3 adds a narrow, passive historical identity foundation for sanitized Phase 1A historical capture events. The goal is to make future historical duration and recurrence work testable without declaring authoritative incident resolution, changing consumer workflows, enabling historical reads, or adding database schema.

## Existing Logic Review

- LP053.2 sanitizer output in `historyCaptureEnvelope.js` is reused because it already allow-lists safe report fields, removes device identifiers, removes free-form detail, excludes exact unrestricted coordinates, rejects unsafe synthetic hazard IDs, and creates three-decimal generalized location buckets.
- Existing envelope creation is reused and extended only by adding an internal `identity` object after sanitization.
- Existing idempotency logic remains separate. It still prevents duplicate writes of the same captured event and is not reused as observation, incident-candidate, or recurrence identity.
- Certified crossing IDs are reused as the strongest crossing location identity because they are stable and safe.
- Runtime road-hazard clustering, hazard lifecycle matching, and clear matching remain separate because they are active-product behavior and can include runtime-only state that should not define passive historical identity.
- Historical recurrence and duration prototypes remain separate because LP053.3 does not calculate duration, prediction, or authoritative recurrence outcomes.
- Active incident audit helpers remain separate because LP053.3 is sidecar-only and must not modify protected systems.

## Identity Contract Version

The identity contract version is `historical_identity_v1`.

The canonical module is `js/history-capture/historyIdentity.js`. It exports pure deterministic helpers only. The helpers do not fetch, write, touch DOM, access storage, call Supabase, mutate input, or log raw input.

## Report, Observation, Incident Candidate, and Recurrence

- **Source report identity** is the safe server-generated public report ID when LP053.2 already allows it. Missing source report IDs are valid. Unsafe device-derived hazard IDs are excluded and source report IDs are not treated as incident IDs.
- **Location identity** is the stable place where evidence was observed. Crossings prefer certified crossing IDs. Road hazards use structured intersection evidence first, then road plus generalized bucket, then generalized bucket.
- **Observation identity** identifies one captured piece of evidence. It is deterministic from safe event/envelope fields, sanitized source report ID when available, category, observed timestamp, location key, lifecycle, and source.
- **Incident candidate identity** groups observations that may belong to one continuous episode. It uses location, condition family, and a fixed UTC epoch time-window bucket. It is a candidate key only, not truth.
- **Recurrence identity** groups separate episodes at the same place and condition family. It intentionally excludes the incident time-window bucket and does not imply prediction or causation.

## Location Identity Hierarchy

### Crossings

1. Certified crossing ID, normalized and stored as `crossing:<id>`.
2. Safe structured fallback evidence only if no certified crossing ID is available.

Crossing names and railroad names are descriptive only. Coordinates never override certified crossing ID identity.

### Road Hazards

1. `intersection:<state>:<county>:<normalized-intersection>` when county/state and structured intersection are available.
2. `road-bucket:<state>:<county>:<normalized-road>:<location-bucket>` when county/state, road identity, and generalized bucket are available.
3. `geo-bucket:<state>:<county>:<location-bucket>` when county/state and generalized bucket are available.
4. `geo-bucket:<location-bucket>` when only a generalized bucket is available.
5. No location key when evidence is insufficient.

Location strength is internal-only:

- `certified` for crossing ID identity.
- `structured` for county/state plus intersection identity.
- `approximate` for road/generalized-bucket and generalized-bucket identities.
- `insufficient` when no safe key should be produced.

A whole county may provide scope, but it is not treated as a precise incident location by itself.

## Token Normalization

LP053.3 uses narrow history-only token normalization: trim, lowercase, collapse separators, remove punctuation that does not change identity, preserve numbers, and normalize common route spacing such as `US90`, `US 90`, and `US-90` to `us-90`. It does not rewrite consumer copy, fuzzy match roads, guess intersections, infer counties, or merge unrelated roads.

## Condition-Family Normalization

Implemented internal condition families are:

- `rail-crossing-obstruction`
- `flooding`
- `construction`
- `disabled-vehicle`
- `debris`
- `livestock`
- `road-closure`
- `lane-closure`
- `crash`
- `other-hazard`
- `unknown`

Crossing blocked and clear observations share `rail-crossing-obstruction` while lifecycle stays separate as `active` or `clear`. Road-hazard clear events retain or derive the original hazard family only when the sanitized category/subtype/report type safely carries it. If the family is unavailable, the family is `unknown` and no incident candidate or recurrence key is generated.

## Candidate Time-Window Policy

Fixed deterministic UTC epoch buckets are used from `observedAt` only:

| Condition family | Candidate window |
| --- | ---: |
| `rail-crossing-obstruction` | 120 minutes |
| `disabled-vehicle` | 120 minutes |
| `livestock` | 120 minutes |
| `debris` | 120 minutes |
| `crash` | 120 minutes |
| `other-hazard` | 120 minutes |
| `flooding` | 360 minutes |
| `lane-closure` | 360 minutes |
| `road-closure` | 360 minutes |
| `construction` | 1440 minutes |
| `unknown` | no candidate key |

These are grouping windows only. They are not durations, predictions, end times, or lifecycle decisions. LP053.3 does not use `expires_at`, local-day buckets, report counts, device identity, or automatic closure.

## Identity Output Contract

The sanitized Phase 1A envelope carries `identity` next to the minimized `report` payload:

```js
{
  version: 'historical_identity_v1',
  sourceReportId: 'safe-example-report-id',
  locationKey: 'intersection:tx:example-county:main-st-and-us-90',
  locationStrength: 'structured',
  conditionFamily: 'flooding',
  lifecycleState: 'active',
  observationKey: 'observation:00000000',
  incidentCandidateKey: 'incident-candidate:11111111',
  recurrenceKey: 'recurrence:22222222',
  candidateWindowMinutes: 360
}
```

Keys are internal and are generated only from sanitized report fields plus envelope-controlled event type and observed timestamp. Identity does not duplicate raw reports and does not include device IDs, unsafe synthetic IDs, free-form detail, exact coordinates, secrets, or unknown raw object fields.

## Relationship to Idempotency

- `idempotencyKey` prevents duplicate writes of the same captured historical event.
- `observationKey` identifies one captured observation.
- `incidentCandidateKey` groups likely same-episode observations within a documented candidate window.
- `recurrenceKey` groups separate episodes at the same place and condition.

These concepts remain distinct and are not collapsed into one hash.

## Privacy and Fail-Open Behavior

LP053.3 inherits LP053.2 payload minimization. Identity is derived only after sanitization. If identity derivation fails, a safe incomplete identity object is returned. If envelope or writer work fails, capture remains fail-open and normal report submission stays protected. Raw reports are never used as identity fallback.

## Audit Helper

Run this passive local-only helper in the browser console:

```js
window.gridlyLp0533HistoricalEventIdentityAudit?.()
```

The audit checks module availability, versioning, sanitizer activity, privacy exclusions, crossing certified identity preference, crossing label independence, road intersection and generalized-bucket support, location-strength classification, condition-family normalization, lifecycle separation, distinct observation/candidate/recurrence/idempotency concepts, create/clear linkage compatibility, confirmation grouping, later recurrence separation, missing timestamp handling, insufficient location handling, fail-open behavior, and disabled historical reads/UI. It returns no raw generated keys.

## Tests

Focused LP053.3 coverage lives in `tests/history-capture/historyIdentity.test.js`. Existing LP053.2 payload minimization, envelope, idempotency, writer, and Phase 1A capture tests remain applicable.

## Known Limitations and Non-Goals

LP053.3 does not perform authoritative incident resolution, prediction, causation, duration calculation, automatic incident closure, historical reads, historical UI, business exports, analytics APIs, database migrations, Supabase schema changes, or production record mutation. Remaining live Supabase deployment status is unknown because this milestone does not connect to or modify live Supabase.

Historical reads, consumer historical UI, prediction, duration, and business exports remain disabled because this milestone is only a privacy-safe identity foundation for future internal historical intelligence work.
