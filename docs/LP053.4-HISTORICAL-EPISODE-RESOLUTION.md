# LP053.4 Historical Episode Resolution Foundation

## Purpose
LP053.4 adds an internal-only, deterministic foundation for resolving sanitized historical observations into evidence-based episode candidates. It supports future questions about first observed evidence, last active evidence, compatible clear evidence, observed duration bounds, unresolved candidates, and recurrence separation without claiming absolute incident truth.

## Core Concepts
- **Observation**: one sanitized historical evidence record with an observation key, lifecycle state, observed time, source report ID when safe, and LP053.3 identity fields.
- **Incident candidate**: the LP053.3 candidate grouping key. It is a deterministic grouping window, not authoritative production incident truth.
- **Episode candidate**: the LP053.4 resolved internal object derived from one incident candidate key and deduplicated observations.
- **Recurrence**: a stable location and condition-family relationship across candidate windows. Recurrence identity is preserved but is not used to merge separate episode candidates.

## Resolution Contract Version
The resolver emits `historical_episode_resolution_v1`. The deterministic `episodeCandidateId` is derived from the resolution version and the incident candidate key. It is not a permanent production incident ID.

## Observation Normalization
`normalizeHistoricalObservation()` accepts sanitized Phase 1A envelopes or already-normalized observations. It copies only safe fields from sanitized reports and identity objects: identity version, observation key, incident candidate key, recurrence key, location key/strength, condition family, lifecycle state, report type, source, confidence, observed time, optional received time, and safe source report ID.

The normalizer does not use raw device identifiers, exact coordinates, free-form detail, unsafe synthetic IDs, or raw fallback records. Invalid timestamps remain unavailable and reduce resolution rather than being replaced by `receivedAt`.

## Grouping Policy
`groupHistoricalObservationsByCandidate()` groups only by valid `incidentCandidateKey` and valid `observedAt`. Observations are never forced into an episode based solely on recurrence key, location key, condition family, source report count, or device identity. Sorting is deterministic by observed time and safe stable fields.

## Deduplication Policy
`deduplicateHistoricalObservations()` suppresses obvious duplicate captured events using this hierarchy:
1. `observationKey` when present.
2. Safe source report ID plus event type, observed time, and lifecycle state.
3. Event type, incident candidate key, location key, condition family, observed time, and lifecycle state.
4. Missing-key observations are retained conservatively.

This removes exact duplicate captures but preserves distinct confirmations that have different observation keys.

## Resolution States
- `active_observed`: active evidence exists and no compatible clear follows the latest active evidence.
- `clear_observed`: compatible clear evidence occurs at or after the most recent active evidence.
- `conflicting_evidence`: active evidence appears after clear evidence, clear evidence is ordered before active evidence, or active and clear evidence cannot be resolved conservatively.
- `unresolved`: reserved for future policy where identity and observations are valid but no reliable clear evidence exists.
- `insufficient_evidence`: identity, location, active evidence, or timestamp evidence is too weak to resolve safely.

The resolver avoids terms such as definitely active, definitely cleared, true, false, or confirmed end.

## Crossing Clear-Linkage Policy
A crossing clear may link when the incident candidate key, location key, condition family, and valid ordered timestamps match. Active -> clear resolves to `clear_observed`. Active -> clear -> active remains `conflicting_evidence`; the clear observation is retained and no evidence is deleted.

## Road Hazard Clear-Linkage Policy
A hazard clear may link only when the incident candidate key, location key, known condition family, sufficient location identity, and valid ordered timestamps match. Approximate road-bucket identity can resolve with moderate linkage confidence when the family is known. Unknown-family hazard clears do not resolve a specific hazard episode and do not guess the family.

## Orphan-Clear Policy
A clear observation with no prior active evidence is preserved as an insufficient-evidence episode candidate when it has a candidate key. The resolver does not fabricate an active start or calculate duration.

## Conflicting-Evidence Policy
Sequences such as active -> clear -> active within the same incident candidate become `conflicting_evidence`. LP053.4 does not split that sequence unless LP053.3 already assigned the later active observation a different incident candidate key.

## First and Last Observation Semantics
The resolver calculates first/last observed evidence, first/last active evidence, and first/last clear evidence using valid UTC `observedAt` values only. It does not convert to consumer local time and does not derive event times from `expires_at` or visibility expiry fields.

## Duration Bounds
Durations are conservative observed bounds, not exact physical incident durations.

- **Observed active span**: first active observation to last active observation.
- **Lower bound**: same as observed active span when active evidence exists.
- **Upper bound**: first active observation to the first compatible clear after the last active observation.
- **Incomplete duration**: unresolved or conflicting candidates retain active span when available but do not complete a duration.

The first observed report is not guaranteed to be the physical start, and the clear observation is not guaranteed to be the exact physical end. `expires_at` is visibility metadata and is not used as an incident end time.

## Confidence Model
Linkage confidence values are `strong`, `moderate`, `weak`, and `insufficient`. Strong linkage requires certified crossing or structured road/intersection identity, matching candidate/family, and valid ordering. Moderate linkage covers approximate location with matching candidate/family and valid timestamps. Weak covers sparse or conflicting evidence. Insufficient covers missing candidate key, unknown family, insufficient location, or unsafe ordering.

Duration confidence values are `high`, `moderate`, `low`, and `unavailable`. High requires stable location identity, known condition family, multiple active observations, compatible clear evidence, no later active evidence, and valid timestamps. Moderate covers one active plus one compatible clear with valid identity. Low covers sparse, weak, or conflicting evidence. Unavailable covers missing identity, unknown required family, invalid timestamps, or insufficient location.

## Privacy Guarantees
LP053.4 uses only sanitized evidence and LP053.3 identity fields. It does not expose raw device IDs, exact coordinates, free-form details, unsafe synthetic IDs, raw fallback payloads, or production database records.

## Fail-Open Behavior
Resolver errors return safe empty or insufficient-evidence results. Historical sidecar failure remains isolated from report submission. No Supabase calls, fetches, storage writes, DOM changes, or report submissions are performed by the resolver or audit helper.

## Audit Helper
Run `window.gridlyLp0534HistoricalEpisodeResolutionAudit?.()` in the browser console. It uses fake in-memory fixtures only and returns pass/fail fields for resolver availability, normalization, duplicate suppression, candidate grouping, crossing/hazard clear linkage, unknown-family safety, conflicts, orphan clears, duration bounds, confidence classifications, privacy preservation, fail-open behavior, and disabled historical reads/UI.

## Historical Engine Integration Status
Integration with the existing historical intelligence prototype is deferred. LP053.4 adds a pure resolver and browser-loaded sidecar so a later adapter can feed resolved episode candidates into the prototype without enabling historical reads, UI, recurring time-of-day patterns, local-time bucketing, API exposure, or consumer insights.

## Known Limitations and Non-Goals
LP053.4 does not predict future conditions, claim physical start/end times, expose consumer historical UI, enable historical reads, add migrations, modify public report inserts, calculate local-time patterns, produce business exports, or connect to live Supabase. Remaining live Supabase deployment status is intentionally unknown in this milestone.

## Tests
