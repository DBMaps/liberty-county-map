# LP053.4B — Unified Active Community State Repair

## Screenshot contradiction
After clearing one blocked-crossing report and one disabled-vehicle report, several live surfaces disagreed: Alerts became quiet, while the top Awareness Brief, a road-hazard marker and popup, and Location Context still presented active community evidence.

## RCA findings
Gridly had multiple active-state readers: raw `activeReports`, raw `activeHazards`, lifecycle-filtered awareness rows, unified incident snapshots, marker-layer reuse, popup HTML bound to older markers, and Location Context reads from `getUnifiedIncidents()`.

## Canonical active community state contract
`gridlyGetCanonicalActiveCommunityState()` is the consumer-current-state selector. It reads normalized current reports and hazards, applies existing lifecycle helpers, applies active county and selected awareness-area context, and returns deterministic plain runtime data: revision, active records, crossing records, road-hazard records, count, county, awareness area, and generation time.

## Lifecycle filtering policy
The canonical state excludes cleared, `hazard_cleared`, recently-cleared, expired, inactive, historical, stale, and lifecycle-suppressed rows. It preserves active blocked crossings and active road hazards, with road-hazard clear suppression and latest crossing state resolution delegated to existing production helpers.

## Canonical revision
The revision includes selected county, selected awareness area, active count, stable active record identities, lifecycle state, type/category, and lifecycle timestamps. It is included in authoritative incident and marker-render signatures so clear correctness does not wait for a minute bucket.

## Unified incident cache invalidation
The authoritative unified incident snapshot key includes the canonical community revision. Clear containment also nulls the cached snapshot before rendering, so `getUnifiedIncidents()` rebuilds from the newest canonical active community state.

## Marker render reuse repair
`renderUnifiedIncidents()` includes canonical active revision/count in its reuse signature and blocks reuse when the active count decreases. The unified marker layer is cleared before rebuild when canonical contents change.

## Popup invalidation and rebinding
Road-hazard clear containment closes any open popup, clears the authoritative snapshot, rebuilds unified incident markers, and refreshes crossing markers. Crossing clear success closes/rebuilds affected popup state and forces crossing render state changes while preserving crossing inventory identity.

## Location Context migration
`updateNearestContext()` now selects nearest active community issues from the canonical active state instead of raw active unified incidents, preventing recently-cleared or stale cached incidents from contributing to “active issue nearby.”

## Active count parity
Alerts, Awareness Story, Travel Brief Community, Community Pulse model inputs, unified community incidents, marker eligibility, popup audit counts, and Location Context now share or synchronize to canonical active count semantics.

## Clear refresh sequence
Crossing clear: persist the clear row, normalize local clear state into `activeReports`, invalidate unified incident snapshot, close popup, force crossing marker refresh, rerender unified incidents, update Location Context, then allow the post-submit/shared-report refresh path to re-run canonical state.

Road-hazard clear: persist the `hazard_cleared` row, capture passive history, apply local clear containment to `activeHazards`, close popup, invalidate unified incident snapshot, rerender unified incidents, force crossing refresh, update Alerts/Pulse/Daily Habit/Location Context, then run the background authoritative shared-report refresh and reapply containment.

## Race prevention
The canonical revision is derived from active lifecycle contents. Older active snapshots with a different canonical revision cannot satisfy the authoritative cache key after a clear, and render reuse cannot preserve stale layers after a count decrease.

## Duplicate clear root cause and repair
Duplicate inserts were structurally possible because popup/direct/delegated invocations could call the same clear submit function before the first insert completed. LP053.4B adds narrow in-memory in-flight guards and passive counters for crossing clears and road-hazard clears. The guards suppress rapid duplicate submissions while resetting after success or failure.

## Official/weather independence
Official roadway and weather evidence remain outside the community selector. Clearing community reports quiets only community contribution; DriveTexas and weather evidence can still elevate official/weather rows and top status.

## Live audit design
`window.gridlyLp0534bUnifiedActiveCommunityStateAudit?.()` is read-only. It performs fixture/contract checks and live runtime parity inspection of canonical count, Alerts count, Awareness count, Travel Brief classification, Community Pulse count, active map markers, active popup copy, Location Context count, unified active count, render revision, stale findings, and duplicate-clear diagnostics. It performs no fetches, writes, storage writes, report submissions, or Supabase calls.

## Browser certification plan
Start quiet, create one blocked crossing and one disabled vehicle, verify all active surfaces agree, clear each once, verify quiet community state across Alerts, top Awareness, Travel Brief, Community Pulse, markers, popups, and Location Context, refresh, switch awareness area and return, then rerun the LP053.4B audit and confirm one action/one submit counters.

## Tests
Focused static coverage verifies the canonical selector, revision, cache invalidation, marker-render signature, popup/context audit fields, duplicate-clear guards, and LP053.4A compatibility. Existing relevant LP053.4A coverage remains.

## Protected systems
No Supabase schema, migrations, production records, report expiration policy, hazard duration policy, category definitions, report form, ZIP personalization, onboarding, Route Watch, Destination Intelligence, DriveTexas authority rules, weather authority rules, PWA lifecycle, LP053.2 payload minimization, LP053.3 historical identity, LP053.4 episode resolution, historical reads/UI/aggregation, exports, notification architecture, or visual styling policy was intentionally changed.

## Known limitations
This change adds deterministic runtime repair and passive certification, but real browser certification with live marker/popup surfaces is still required before LP053.5 resumes.

## Why LP053.4A was insufficient
LP053.4A certified only fixture/model behavior. It did not inspect live marker layers, bound popups, Location Context, top Awareness, rendered Alerts parity, active marker counts, refresh/area-switch behavior, or duplicate clear submissions.

## LP053.5 pause
LP053.5 remains paused until the live browser certification plan passes with `safeToResumeLp0535: true`.
