# LP053.4C — Immediate Clear State Convergence

## Browser evidence

LP053.4B browser validation showed that the full reload path was correct, but the local post-clear runtime path could remain stale until refresh. After a crossing blockage and a flooding hazard were cleared, some surfaces became quiet while Top Awareness, Location Context, and the active issue count still presented the flooding condition.

## Why LP053.4B was incomplete

LP053.4B centralized active community reads, but the road-hazard clear path still performed local marker/awareness containment with a narrower coordinate/type filter and then ran multiple refresh calls. The immediate path could remove one marker-oriented candidate while leaving lifecycle evidence or cached awareness models inconsistent with the state that `loadSharedReports()` rebuilds from active rows plus recent `hazard_cleared` rows.

## Local clear evidence parity

`gridlyApplyImmediateClearLifecycleConvergence()` is the central immediate convergence helper. It accepts a clear kind and normalized clear row, updates the same local lifecycle inputs used by reload, invalidates current-state caches, recomputes canonical state, rebuilds unified incidents/markers, and refreshes consumer surfaces once.

### Crossing clears

The helper replaces any in-memory report for the crossing with the normalized clear row. The crossing inventory remains untouched, but the latest local crossing lifecycle state becomes clear before canonical state and marker/popup rendering run.

### Road-hazard clears

The helper combines current `activeHazards`, `recentlyClearedRoadHazards`, and the normalized local `hazard_cleared` row, then reruns `gridlyFilterRoadHazardsByLatestLifecycle()` and `gridlyFilterRecentlyClearedRoadHazardsForVisibility()`. This reuses reload semantics instead of inventing a weaker coordinate-only suppression rule.

## Canonical recomputation order

The implemented order is:

1. Persist clear.
2. Normalize the local clear row.
3. Update `activeReports`, `activeHazards`, and/or `recentlyClearedRoadHazards`.
4. Re-run authoritative lifecycle filtering.
5. Invalidate current-state caches and reuse signatures.
6. Recompute canonical active community state.
7. Invalidate unified incident snapshot state.
8. Rebuild unified incidents.
9. Rebuild markers/popups.
10. Render Alerts and current awareness surfaces.
11. Refresh Community Pulse, Travel Brief, Top Awareness, mobile mirrors, Location Context, and active counts.

## Unified incident convergence

Unified incidents are rebuilt after the lifecycle collections and canonical state are updated, so a cleared flooding row can remain as recently-cleared evidence without contributing to active incidents, markers, popups, or counts.

## Awareness and Location Context convergence

The convergence helper clears refresh-cycle caches and Community Pulse/awareness reuse signatures before running the refresh chain. Location Context is updated after canonical recomputation, so prior flooding copy and “1 active issue nearby” cannot be reused when canonical active count is zero.

## Generation and race protection

Each convergence increments `gridlyLp0534cClearConvergenceGeneration`. A `loadSharedReports()` invocation that started before a clear generation advanced is suppressed unless it is the clear’s own background confirmation path. This prevents older active snapshots from overwriting a newer cleared runtime state.

## Immediate-versus-authoritative diagnostics

The runtime tracks safe counters and signatures: convergence starts/completions, authoritative refresh completions, mismatch count, last immediate/authoritative canonical counts, last generation, and stale generation suppressions. It compares active counts and safe crossing/hazard keys without exposing raw report payloads, device IDs, exact coordinates, or credentials.

## Audit helper

`window.gridlyLp0534cImmediateClearStateConvergenceAudit?.()` is passive and read-only. Contract fields describe whether required helpers and ordering are present. Live fields inspect current canonical, unified incident, alert, marker, popup, and Location Context counts. The audit fails when live counts disagree, stale awareness/context/popup/marker evidence remains after canonical count reaches zero, or immediate and authoritative counts diverge.

## Tests

Focused LP053.4C structural tests cover helper presence, authoritative lifecycle reuse, crossing replacement, cache invalidation ordering, unified rebuild, Location Context update, generation suppression, mismatch diagnostics, and preservation of the LP053.4B audit.

## Known limitations

LP053.4C still requires browser certification because visual correctness on real mobile portrait surfaces must be confirmed with live interactions. LP053.5 remains paused until the LP053.4C audit and no-refresh browser workflow pass. Browser refresh must never be required for correctness; it should only confirm that the immediate runtime state already matches authoritative reload state.
