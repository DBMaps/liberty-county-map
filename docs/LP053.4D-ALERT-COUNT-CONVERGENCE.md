# LP053.4D — Alert Count Convergence

## Root cause

The lifecycle repair from LP053.4C correctly reduced the canonical active community state, unified incidents, markers, and popups to zero immediately after clear. Alerts could still report one active alert because its count helper and snapshot had an independent fallback path: `activeReports` plus road-hazard fallback rows, the active render context, `getAlertsSurfaceSnapshot()`, and `window.__gridlyLatestAlertsForRender`.

That meant a cleared row could remain visible to the Alerts ownership chain even after canonical active state had converged.

## Repair

LP053.4D repairs ownership rather than forcing `alertActiveCount` to equal `canonicalActiveCount`. The Alerts active-row helper now starts from canonical active community state. The Alerts snapshot also uses canonical active community records before any legacy fallback. Legacy context, snapshot, render-cache, and localized-intelligence fallbacks remain available only when canonical state is unavailable.

## Audit

`window.gridlyLp0534dAlertCountConvergenceAudit?.()` is passive, read-only, performs no fetches, performs no writes, and performs no storage writes. It reports canonical, Alerts, unified incident, marker, and popup counts, verifies Alerts renderer/audit ownership flags, checks the actionable Alerts DOM against rendered state, and fails if Alerts disagrees with canonical, unified incidents, or markers.

## Regression tests

The focused structural test covers one flooding clear, one crossing clear, combined flooding and crossing clear, renderer rebuild ordering, alert audit exposure, hidden rows, header/empty rows, cached alert reuse, refresh-free convergence, and regression ownership checks by asserting the repaired source chain and mismatch failure conditions.
