# V344.1 — Alert Location Ownership Loop Audit

## Goal

V344.1 is an audit-only milestone for the browser hang reported immediately after the V344 road-hazard alert location ownership fix. It does not attempt another alert wording fix.

Reported symptom:

- Browser tab hang / `RESULT_CODE_HUNG` after V344.

Suspected path:

```text
Alert creation
→ location ownership bridge
→ alert snapshot shaping
→ alert refresh
→ location ownership bridge again
→ alert snapshot reshaping again
→ refresh/render loop
```

## Scope Guard

This milestone intentionally does **not** change:

- alert wording
- alert rendering
- marker rendering
- lifecycle rules
- V343 suppression ordering
- Supabase behavior
- Route Watch
- crossings
- DriveTexas work
- official-source architecture
- production user-facing behavior

## Runtime Helper

V344.1 adds:

```js
window.gridlyAlertLocationOwnershipLoopAudit?.()
```

The helper returns the V344.1 audit contract:

```js
{
  available: true,
  policyVersion: "V344.1",
  productionBehaviorChanged: false,
  pageHangObservedByUser: true,
  bridgeInvocationCount,

```

Additional `diagnostics` are included for bridge invocation delta, max bridge depth, recent bridge invocations, refresh deltas, source mutation samples, and per-alert idempotency rows.

## Audit Questions Answered

The audit explicitly answers:

1. Did V344 introduce non-idempotent alert rewriting?
2. Is the same alert/card being rewritten repeatedly?
3. Is the location bridge mutating source records?
4. Is alert snapshot shaping triggering refresh/render loops?
5. Is V344 safe to merge?

Unless browser validation proves otherwise, the helper returns:

```js
safeToMergeV344: false
```

## Loop-Safety Diagnostics

The audit records and reports:

- number of V344 bridge invocations
- number of alert-card snapshot rewrites observed during audit shaping
- whether shaping is idempotent across repeated calls
- whether the same alert identity appears repeatedly
- whether source arrays or alert objects mutate during shaping
- whether refresh counts change during shaping
- whether recursive bridge depth is detected
- whether candidate pool scans may be expensive

## Manual Browser Validation

After applying V344.1, reload the page and run:

```js
window.gridlyAlertLocationOwnershipLoopAudit?.()
window.gridlyBackgroundLoopAudit?.()
window.gridlyRefreshBreakdownAudit?.()
window.gridlyRefreshAudit?.()
```

Expected result: the audit should identify whether V344 caused recursion, repeated non-idempotent reshaping, source mutation, refresh amplification, or expensive repeated scans. V344 should remain unsafe to merge until the hang path is ruled out.
