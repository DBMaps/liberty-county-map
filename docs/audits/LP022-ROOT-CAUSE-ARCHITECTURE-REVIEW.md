# LP022 Root-Cause Architecture Review

Investigation-only review for `LP022-AUTHORITATIVE-LOCATION-REPAIR`. No production repair, CSS, report-submission, Supabase, DriveTexas ingestion, marker, focus, clearing, crossing workflow, Story Engine, or runtime-refresh changes are made by this report.

## Executive findings

1. Ordinary community flooding reports do not have useful consumer location when the submission path cannot trust a road label. The standard saved report row carries coordinates, a report type, county metadata, and `crossing_name` equal to the hazard label when no `locationName` is supplied. Useful location only appears if the local road-aware placement/snapping path has a `selectedRoadName` and `gridlyBuildRoadHazardSubmissionLocationPayload()` accepts it.
2. The live report path does not generate directional-place phrases such as `1 mile north of Goodrich`. The only exact occurrence of that phrase is an LP022 audit sample, not normal submission logic.
3. `gridly_structured` metadata is generated by live submission only inside the `detail` string via `appendGridlyStructuredMetadata()`. It can preserve accepted road metadata, county metadata, and submitted coordinates; it does not create community, nearest-place, or direction/distance fields.
4. The official DriveTexas path currently promotes advisory/body text as location because `gridlyOfficialSituationLocation()` reads `locationDescription`, `sourceLocationDescription`, `normalizedDescription`, `locationName`, `location`, and finally `description` as `detail`, then `buildGridlyOfficialSituationAlert()` copies that result into every canonical location field.
5. The DriveTexas provider normalizer currently exposes route, description, coordinates, IDs, category, and dates. It does not expose a deterministic nearby-place or direction-distance field. If the live provider record only contains `US 59` plus coordinates and advisory prose, Gridly should display `US 59` only as the location line unless a source-specific DriveTexas adapter proves a stronger provider field.
6. The architecture is conflating location description, incident description, advisory text, lane status, project narrative, route name, county context, and presentation text across several generic helpers. Repeated local fixes add candidate fields and fallbacks without ownership boundaries, which repairs one surface while regressing another.

### Community geospatial capabilities invoked today

* Reverse geocoding: **No evidence in the standard submission path.**
* Nearest road lookup: **Yes, conditionally**, through road-aware placement/snapping and `resolveNearestRoadName()` backed by local roadway/crossing datasets.
* Nearest community lookup: **No.**
* Nearest known place lookup: **Not for ordinary consumer report location.** `resolveNearbyKnownLocation()` scans crossings and returns a road-like crossing label, not a community/nearby-place phrase.
* Crossing lookup: **Yes as road fallback**, not as a user report's nearest-place string.
* Coordinate-to-county lookup: **Yes**, by supported county bounds only.
* Distance/direction calculation: **No user-facing direction-distance phrase is generated for ordinary flooding reports.**

Therefore, `1 mile north of Goodrich` is not produced by a standard community report today.

## Part 2: origin of the Goodrich phrase

Search results show `1 mile north of Goodrich` only in LP022 audit samples in `js/app.js`. The nearby `Goodrich` references are county awareness area names and audit mismatch heuristics, not submission logic or provider-normalized fields.

Conclusion: this was never a valid expectation for an ordinary user-created flooding report. Without a new geospatial enrichment capability, Gridly can truthfully produce only:

* accepted road label from local road-aware placement, e.g. `US 59`, when `selectedRoadName` is trusted;
* county/awareness context, e.g. `Polk County` or selected area context, when no trusted road label exists;
* coordinates internally, not as consumer location copy.

`Near Livingston` or `1 mile north of Goodrich` require a trusted community/place lookup and, for the latter, a deterministic distance/bearing-to-place formatter that does not exist in the ordinary submission path.

## Part 3: official provider semantics

### Current DriveTexas normalized object semantics

| Field | Semantic type | Notes |
|---|---|---|
| `id` | identifier | From `GLOBALID`, `globalId`, `id`, `event_id`, or generated fallback. |
| `provider`, `providerId`, `sourceTrace` | identifier | Provider provenance. |
| `category`, `title`, `resolvedHeadline` later | incidentDescription | `category` is normalized condition type; generated title is category plus route. |
| `description` | advisory / incidentDescription / projectDescription / laneStatus | Provider prose. It is not guaranteed to be location. |
| `routeName` | route | From `route_name`, `routeName`, `roadway`, `road`, or `highway`. |
| `latitude`, `longitude` | coordinates | From GeoJSON point/line midpoint or coordinate fields. |
| `startTime`, `endTime` | dates | Schedule/freshness window. |
| `county` if present raw only | county | The provider normalizer does not currently expose it on normalized records. |

### Exact field that produced invalid official location text

The invalid strings:

* `MAIN LANES not affected...`
* `Construction of safety improvement projects...`
* `Alternating lanes closed...`
* `Motorists should expect delays...`
* `INSTALL CABLE MEDIAN BARRIER`

were promoted from provider advisory/prose fields that enter Gridly as `description` and then become `sourceLocationDescription` or `normalizedDescription` in `buildGridlyOfficialSituationAlert()`. The immediate architecture bug is two-step:

1. `gridlyOfficialSituationLocation()` treats `record.description` as a possible location detail after checking other location-like fields.
2. `buildGridlyOfficialSituationAlert()` copies the resulting `location` into `canonicalDisplayLocation`, `canonicalLocationPhrase`, `authoritativeLocationLabel`, `strongestLocationLabel`, `resolvedLocationLabel`, `displayLocation`, `nearbyLocationPhrase`, `knownLocation`, `locationName`, and `subtitle`.

Because LP022 gives `canonicalDisplayLocation` highest priority, advisory/body text copied into canonical location fields outranks safer route-only display.

### Does stronger official location data exist?

From the checked normalizer contract, no deterministic direction-distance or nearby-place field is exposed. The normalizer reads route from route/roadway keys, body text from description/summary/title/headline, coordinates from geometry or coordinate fields, and dates/ID. If the live Construction/Lane Closure records contain only `US 59`, coordinates, and advisory text, Gridly should state only `US 59` in the location line and place advisory/project/lane copy in details. A stronger location should be accepted only if a DriveTexas schema-specific field is verified as location, such as an explicit provider `locationDescription` whose contract excludes lane/advisory/project prose.

## Part 4: current location-ownership architecture map

| System | Current ownership | Problem |
|---|---|---|
| Report submission | Captures coordinates, optional road-aware `selectedRoadName`, county metadata, and embeds `gridly_structured` in `detail`. | Does not own nearest community/place/direction-distance. County is context, not precise location. |
| `normalizeReports()` | Parses `detail` metadata and promotes location-like fields into normalized records. | Pass-through only; cannot create missing specificity. Field names are broad and untyped. |
| Canonical road-hazard incidents | Groups reports and derives/preserves labels through multiple V322/LP helpers. | Canonical incident model does not own a typed location contract before presentation. |
| DriveTexas provider normalization | Converts raw provider records to route, description, category, coordinates, dates, ID. | Does not classify prose semantics; `description` later becomes location. |
| Official situation creation | Builds official alert records and canonical location fields. | Owns the current severe bug by promoting `description`-derived detail to canonical location. |
| LP020 helper | `gridlyLp020BestAvailableLocation()` acts as a broad best-available fallback. | Generic fallback chain over heterogeneous records risks source-specific mistakes. |
| LP021 adapter | `gridlyLp021ResolvedLocationPresentation()` delegates to LP022 and records traces. | Useful as presentation adapter/audit, but it depends on upstream candidate truthfulness. |
| LP022 resolver | Prioritizes canonical and location-like fields, rejects many generic/technical values. | Still generic; if upstream canonical fields are polluted, priority makes pollution authoritative. |
| Alert cards | Consume LP021/LP022 model and render location line. | Presentation cannot distinguish route/location/advisory once canonical field is polluted. |
| Community popup | Resolves authoritative location then popup road fallback. | Can display county/road fallback when no source location exists. |
| Crossing popup | Crossing-specific model uses crossing identity. | More source-specific and safer; should remain separate. |
| Official popup | Uses LP021/LP022 on official records. | Renders polluted canonical official location. |
| Travel Brief | Uses its own route/location/description cleaner and DriveTexas lines. | Separate generic logic; another fallback chain that can diverge from alerts/popups. |

Duplicate/fallback patterns include LP020, LP021, LP022, V322 strongest-location helpers, popup road label helpers, Travel Brief cleaners, and official situation location creation. Source-specific knowledge appears inside generic helpers via field names such as `description`, `normalizedDescription`, `locationDescription`, `crossingName`, `countyId`, `roadwayName`, and `routeName`. Presentation text is reused as data when official summary/location is copied into canonical display fields. County/community context is substituted when no precise incident location exists.

Repeated local fixes have regressed because each fix broadens or reprioritizes fallback candidates without a source-owned semantic contract. Once a prose field is mislabeled as canonical location, every downstream resolver faithfully preserves the wrong value.

## Part 5: architecture options

### Option A: one universal resolver over heterogeneous raw records

* Benefits: lowest short-term code movement; one place to tune presentation; compatible with current LP021/LP022 tests.
* Risks: repeats current failure mode; generic field names cannot know provider semantics; upstream pollution remains indistinguishable from true location.
* Migration cost: low.
* Regression risk: high, especially official provider and future weather/crossing fields.
* Compatibility: superficially high but architecturally brittle.
* LP016 impact: likely low if cached, but continued fallback scans can add complexity.
* Testing strategy: large matrix of negative prose cases and per-source fixtures; difficult to prove complete.

### Option B: source-specific canonical location adapters

Each adapter returns:

```js
{
  primaryLocation,
  secondaryLocation,
  precision,
  provenance,
  coordinates,
  safeForConsumerDisplay
}
```

* Benefits: separates community, crossing, DriveTexas, and weather semantics; rejects unsafe prose at source boundary; presentation consumes one typed contract; prevents `description` from becoming location unless a source adapter explicitly classifies it safe.
* Risks: requires migration of existing surfaces to consume the contract; adapters must be carefully specified.
* Migration cost: moderate.
* Regression risk: low-to-moderate if introduced shadow-first and compared with current output.
* Compatibility: good with Gridly architecture because existing source identities already exist and crossing-specific handling is already safer.
* LP016 impact: low if adapters are pure, bounded, and run during normalization/canonical incident creation with no new scans in render loops.
* Testing strategy: per-source fixture suites with field semantic classification, unsafe-prose rejection, route-only official cases, coordinate-only community cases, and consumer contract snapshots.

### Option C: canonical incident model owns location before presentation surfaces

* Benefits: strong central authority; alerts/popups/travel brief all consume identical canonical incident location; best long-term model.
* Risks: larger migration; needs source-specific semantic input anyway; premature centralization can hide source limits if not provenance-rich.
* Migration cost: high.
* Regression risk: medium during migration due to broad surface impact.
* Compatibility: good long term, but intrusive against current protected systems.
* LP016 impact: can be low if done once per incident, but risky if it triggers more canonical rebuild work.
* Testing strategy: canonical incident golden snapshots plus source adapter fixtures and render-consumer snapshots.

### Recommendation

Recommend **Option B now, evolving into Option C later**. The smallest stable architecture is source-specific location adapters that produce a typed, provenance-bearing display contract, followed by presentation surfaces consuming only that contract. Once adapters are stable, canonical incident creation can own the same contract as part of Option C.

Explicit answers:

* Community report location should originate in the community report adapter from submission-time trusted fields: accepted road-aware placement label, trusted future geospatial enrichment, coordinates, and county/awareness context as context only.
* Official location should originate in the DriveTexas adapter from verified provider schema fields: route, explicit provider location fields if proven safe, and coordinates. Advisory/body/project/lane prose must be rejected as location.
* Missing specificity should be represented with `precision: "coordinate_only"`, `"route_only"`, `"county_context"`, or `"none"`, not filled by arbitrary prose.
* Gridly should display roadway fallback for official route-only records (`US 59`) because route is a real provider field. For community reports, county-only should be visibly treated as area context or omitted in the location line depending on product copy; it must not be portrayed as incident-specific.
* Never parse or synthesize location from advisory body, lane status, project narrative, generic incident description, generated title, hazard label, or county/community ID. Do not synthesize direction-distance without a deterministic geospatial feature and provenance.
* Retain LP021 as a thin presentation adapter/audit only. Retain LP022 negative validation/audit logic as guardrails. Retire or demote LP020 broad best-available and LP022 universal resolution as authority once adapters exist. Replace official `description` promotion and any canonical-field pollution with adapter output.

## Part 7: audit quality review

`window.gridlyLp022LiveLocationPipelineAudit?.()` can report `recordsWithDetailLoss: 0` while output remains poor because it measures preservation of the strongest candidate that ever appeared in the current pipeline. If the pipeline never had useful community location, there is nothing to lose. If upstream canonical official location is already polluted by advisory text, preserving it is counted as success.

The audit currently overweights “nothing was lost” and underweights “a useful trustworthy location exists.” Add dimensions:

* `locationAvailableAtSource`
* `locationPrecision`
* `locationProvenance`
* `sourceLimited`
* `unsafeDescriptionRejected`
* `consumerUsefulLocation`
* `fallbackReason`
* `requiresGeospatialEnrichment`
* `canonicalFieldPolluted`
* `routeOnlyAccepted`
* `countyContextOnly`

## Smallest safe next milestone

Create a documentation/test-only location contract milestone before production repair:

1. Define adapter contract and allowed semantics for community, crossing, DriveTexas, and weather.
2. Add fixture/audit tests that classify DriveTexas `description` samples as advisory/lane/project prose, not location.
3. Add fixture/audit tests showing ordinary community flooding with only coordinates has `precision: "coordinate_only"` or `"county_context"` and `requiresGeospatialEnrichment: true`.
4. Shadow-run adapters without changing rendering, Supabase, DriveTexas ingestion, LP016 caching, awareness selection, or CSS.
5. Only after review, replace official canonical location creation to consume DriveTexas adapter output.

## Current LP022 change disposition

* Retain: audit exposure, negative checks for internal identifiers/county IDs/generic hazard labels, and trace tooling.
* Retain temporarily: LP021 delegating through LP022 as a presentation adapter while adapters are absent.
* Revert/replace: any LP022 or official-situation behavior that treats `locationDescription`, `sourceLocationDescription`, `normalizedDescription`, or `description` as authoritative for DriveTexas without a source-specific schema guarantee.
* Replace: universal best-available resolver authority with source-specific adapters, then canonical incident-owned typed location.
* Do not add: resolver fallback fields or new promotions into `canonicalDisplayLocation`.
