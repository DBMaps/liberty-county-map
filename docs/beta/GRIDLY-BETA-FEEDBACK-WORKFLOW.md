# GRIDLY BETA FEEDBACK WORKFLOW

## Status

V900

## Purpose

This workflow keeps beta feedback organized so Gridly can improve based on clear evidence instead of scattered memory, texts, and casual conversations.

## Single Feedback Intake Principle

Use one primary place to capture feedback. Testers may mention issues by text or in person, but the beta operator should still copy the useful details into the chosen tracking place.

Do not let feedback scatter across memory, texts, and casual conversations without capturing it.

## Recommended Feedback Categories

- Bug
- Incorrect location
- Incorrect information
- Installation issue
- Performance issue
- Confusing wording
- Visual issue
- Suggestion
- Safety/trust concern
- General feedback

## Required Information To Collect From Testers

- Tester name or identifier
- Date/time
- Device type
- Browser or install mode
- Awareness area
- What happened
- Expected behavior
- Screenshot if available
- Beta URL/version if known

## Feedback Review Process

1. Capture feedback in the intake place.
2. Assign a category.
3. Decide whether it is a one-off note, a pattern, or an actionable issue.
4. Check whether it affects trust, reporting, installation, onboarding, location correctness, or safety language.
5. Prioritize using the beta issue triage guide.
6. Close the loop with the tester when appropriate.

## How Feedback Becomes An Issue

Feedback should become an issue when:

- It can be reproduced or described clearly.
- More than one tester reports the same confusion.
- It affects trust, safety, installation, location, reporting, or core use.
- It reveals wording that causes users to misunderstand what Gridly is.
- It identifies a visible broken state or strong polish concern.

## How Issues Become Codex Work

Codex work should be narrow and audit-first:

- Summarize the user-facing problem.
- Include device, browser, install mode, screenshot, and beta URL/version if available.
- State priority and why.
- Identify protected systems that must not be changed unless explicitly in scope.
- Ask for documentation-only work when the issue is process or communication related.
- Avoid turning every suggestion into product work.

## How To Close Feedback Loops With Testers

- Thank the tester.
- Confirm the feedback was captured.
- Ask one follow-up question if needed.
- Tell the tester when an issue is fixed, deferred, or being watched.
- Use plain language, not technical release notes.

## What Not To Do

- Do not rely on memory.
- Do not turn every suggestion into immediate product work.
- Do not argue with tester confusion; treat confusion as a valid product signal.
- Do not overpromise fixes or dates.
- Do not change protected systems casually during beta.
- Do not prioritize cosmetic polish over bugs affecting trust, reporting, installation, or location correctness.
