# Gridly Beta Install Experience

## Purpose

This document defines the complete tester install and first-use journey for the Gridly beta. The goal is to make first access feel clear, calm, and reliable without changing application runtime behavior.

## Complete Journey

### 1. Open Beta Link

Testers start from the current beta URL:

<https://dbmaps.github.io/liberty-county-map/?v=beta-clean-4>

Expected result: Gridly opens in the browser without requiring a separate account or app-store download.

### 2. Install Gridly

If the tester wants faster repeat access, they may install or add Gridly to the home screen using their browser's standard install flow.

Expected result: the tester understands that install prompts vary by device and browser, and that browser access remains acceptable.

### 3. Complete Onboarding

The tester reviews onboarding prompts and confirms the basic experience.

Expected result: the tester understands Gridly's mission, beta status, and awareness-first purpose.

### 4. Choose Awareness Area

The tester selects or confirms the area they want Gridly to focus on.

Expected result: the beta experience feels locally relevant.

### 5. Allow Location

When prompted, the tester may allow location access.

Expected result: location-enabled awareness becomes more relevant. If the tester declines, they should still understand how to continue with reduced local context.

### 6. Review Awareness Brief

The tester reviews the Awareness Brief before using deeper features.

Expected result: the tester can quickly understand current nearby conditions and known limitations.

### 7. Submit First Report

When appropriate, the tester submits a first community report.

Expected result: the tester understands that reports should be accurate, safe to submit, and free of private personal information.

### 8. Understand Community Pulse

The tester reviews Community Pulse as an awareness signal, not a guarantee.

Expected result: the tester understands that community activity may be incomplete, delayed, duplicated, or unverified.

### 9. Know Where to Get Help

The tester knows where support guidance and feedback instructions live.

Expected result: the tester can report confusion, install trouble, inaccurate information, or broken flows.

## Common Install Problems

### Install Prompt Does Not Appear

Some browsers do not show an install prompt immediately. Testers should continue using the beta in the browser or use the browser menu to add the page to the home screen when supported.

### Home Screen Icon Opens the Wrong Version

The tester should remove the old icon, reopen the current beta URL, and add it again.

### Location Permission Was Denied

The tester can continue with reduced local awareness. To recover, they may update browser or device location permissions and reload the beta.

### Page Appears Stale

The tester should reload the page, close and reopen the browser tab, or reopen the current beta URL from the invitation.

### Poor Connectivity

Gridly may load slowly or show incomplete information when the connection is weak. Testers should retry when they have a more stable connection.

## Recovery Guidance

Recommended recovery order:

1. Reload the page.
2. Reopen the current beta URL.
3. Close and reopen the browser.
4. Remove any old home-screen shortcut and reinstall from the current beta URL.
5. Confirm device browser and location permissions.
6. Send feedback if the issue continues.

Feedback should include device, browser, time, network condition if known, and screenshots when possible.

## Success Criteria

The install experience is successful when:

- Testers can open the beta URL.
- Testers understand install is optional and device-dependent.
- Onboarding sets awareness-first expectations.
- Location permission behavior is understandable.
- The first Awareness Brief is easy to find and review.
- Testers can submit a safe first report.
- Testers know Community Pulse is contextual, not authoritative.
- Testers know where to get help.
