# GRIDLY BETA OPERATIONS RUNBOOK

## Status

V900

## Purpose

This is Denise's practical operating guide for running the Gridly beta without chaotic expansion. It is checklist-oriented and intended to preserve the current polished product experience.

## Before Inviting Testers

- Confirm the current beta URL.
- Confirm the intended cohort.
- Confirm where feedback will be captured.
- Review known limitations.
- Prepare the welcome message.
- Decide how many testers can be supported this week.

## Inviting Testers

- Invite only the planned cohort.
- Keep the message short and human.
- Tell testers Gridly is for awareness and does not replace official emergency sources.
- Ask testers to report confusion, trust concerns, install issues, and location concerns.

## Tracking Who Was Invited

Track:

- Name or identifier.
- Cohort.
- Date invited.
- Whether they opened Gridly.
- Whether they installed it.
- Whether they provided feedback.
- Follow-up needed.

## Tracking Beta URL/Version

Record the URL/version shared with each cohort. Current beta URL:

## Collecting Feedback

- Capture feedback in one place.
- Add date/time, device, browser/install mode, and awareness area.
- Request screenshots when safe and helpful.
- Mark trust, reporting, install, onboarding, and location issues clearly.

## Reviewing Issues

- Review feedback at least weekly during active beta.
- Assign category and priority.
- Separate bugs from suggestions.
- Watch for repeated confusion.
- Escalate P0/P1 issues.

## Preparing Codex Work

- Prepare narrow requests.
- Include tester evidence.
- State what must not be touched.
- Keep audit-first, patch-second discipline.
- Do not request broad rewrites during beta.

## Testing Fixes

- Test the exact issue reported.
- Test on the device/browser pattern if known.
- Confirm the fix does not change protected systems unintentionally.
- Ask the original tester to retry when useful.

## Deciding Whether To Merge

- Merge documentation/process work after static review.
- Merge runtime changes only when scoped and tested.
- Do not merge if P0/P1 risk is unclear.
- Avoid formatting churn or unrelated changes.

## Communicating Updates

- Use plain language.
- Say what changed for testers.
- Avoid technical details unless needed.
- Remind testers about known limitations when relevant.

## Pausing Rollout

Pause when:

- A P0 is open.
- A P1 affects core trust or a large portion of testers.
- Feedback is too scattered to manage.
- Testers misunderstand emergency limitations.
- The beta link spreads beyond the intended cohort.

## Resuming Rollout

Resume when:

- P0 issues are resolved.
- P1 issues are resolved or clearly bounded.
- Feedback is organized.
- Tester communication is ready.
- Denise can support the next group.

## Weekly Beta Review Checklist

- Current cohort reviewed.
- New testers counted.
- Active testers counted.
- Install and first-open issues reviewed.
- Reporting and location issues reviewed.
- Trust and safety concerns reviewed.
- P0/P1 issues reviewed.
- Feedback response status checked.
- Decision recorded: continue, pause, or expand.

## Beta Expansion Checklist

- No open P0 issues.
- P1 issues resolved or accepted with clear limits.
- Feedback intake is working.
- Current cohort is not creating support overload.
- Liberty / Dayton confidence remains high.
- Next cohort size is controlled.
- Welcome and limitation messages are ready.

## Beta Safety Checklist

- Testers know Gridly is not for emergencies.
- Testers know official sources come first.
- Testers understand community reports may be incomplete.
- Testers are not promised real-time official emergency information.
- High-risk public audiences are not invited yet.
