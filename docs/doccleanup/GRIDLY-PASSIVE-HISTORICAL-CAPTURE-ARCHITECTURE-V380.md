# V380 — Passive Historical Capture Architecture

## 1. Architectural Goals

V380 designs a future passive historical evidence capture architecture for Gridly. It is an architecture-design milestone only and does not approve implementation, migration execution, Supabase deployment, application changes, historical reads, historical writes, history UI, or production integration.

The core product posture is:

> Capture Everything. Show Nothing. Depend On Nothing.

The future capture system should pursue the following objectives:

- **Preserve historical evidence passively:** capture lifecycle evidence generated by normal report and incident activity without requiring users, alerts, awareness, markers, Route Watch, or incident lifecycle logic to depend on the historical system.
- **Keep evidence non-authoritative:** stored evidence must be treated as an audit and analytics input, not the live source of truth for active conditions, routing, alerts, awareness, or public map state.
- **Avoid user-visible behavior:** capture must not introduce history UI, historical reads, public history surfaces, live map changes, marker changes, alert changes, awareness changes, or Route Watch changes.
- **Remain fail-open:** capture failure must never block report submission, report updates, report clearing, incident transitions, alert generation, awareness updates, marker rendering, Route Watch behavior, or DriveTexas-related behavior.
- **Support later validation:** captured evidence should be structured enough to validate completeness, duplicate rates, volume, source provenance, and retention policy without becoming a production dependency.
- **Remain reversible:** the capture system should be disableable, removable, and ignorable without changing the production incident model or user experience.
- **Respect current program boundaries:** because V363 through V379 established historical projection and schema governance readiness without production integration, V380 only defines the next architectural option and does not activate it.

Current state remains unchanged:

- Migration applied: **NO**.
- Supabase changed: **NO**.
- Historical reads: **NO**.
- Historical writes: **NO**.
- History UI: **NO**.
- Production integration: **NO**.

## 2. Architectural Principles

### Fail-open

Historical capture must be best-effort. Any failure in capture transformation, deduplication, persistence, monitoring, or validation must be contained inside the historical capture boundary. The production action that generated the candidate evidence must continue as if capture did not exist.

Required behavior:

- Do not reject a report because historical evidence could not be stored.
- Do not delay alert generation while waiting for historical capture.
- Do not delay awareness, marker, Route Watch, or incident lifecycle updates while historical capture is attempted.
- Do not retry synchronously in a way that affects user-visible latency.
- Do not surface historical capture errors to users as production errors.

### Non-authoritative

Historical evidence must not become the authority for live conditions. Active state remains owned by current production systems such as shared reports, active hazards, unified incidents, alerts, awareness, markers, Route Watch, and DriveTexas-derived behavior.

Historical evidence may later support analytics, audits, retrospective review, recurrence analysis, or duration analysis only after separately approved milestones. It must not be used to decide whether an incident is active, whether an alert should fire, whether a marker should render, or whether Route Watch should react.

### Isolation

Capture must be isolated by boundary, dependency direction, and operational ownership:

- Production systems may emit or expose candidate evidence events, but they must not depend on capture success.
- Capture storage must not be queried by live production flows.
- Capture code must not mutate live incident objects as part of evidence persistence.
- Capture monitoring must observe the capture path without driving live behavior.
- Capture rollout, disablement, retention, and cleanup must be independently controllable.

### Reversibility

The architecture must allow Gridly to disable, pause, delete, or ignore historical capture artifacts without affecting production. Reversibility must be designed before activation, including disablement controls, rollback plans, and documentation that no production behavior depends on historical evidence.

### No Production Dependency

Production must not require historical tables, historical policies, historical storage functions, capture queues, backfills, analytics jobs, or evidence readers. If the historical capture system is absent, offline, disabled, or empty, Gridly should continue to operate with the same user-facing behavior.

## 3. Candidate Capture Sources

This section evaluates candidate capture sources for a future passive system. It recommends a preferred source but does not implement any source.

| Candidate source | Evidence value | Isolation risk | Completeness risk | Recommendation |
| --- | --- | --- | --- | --- |
| Report creation | High. Captures initial user- or system-originated condition evidence at the earliest durable point. | Medium if wired directly into submission flow; low if emitted as best-effort sidecar evidence after primary success. | Low for new reports, but does not capture later lifecycle changes by itself. | Recommended as one input, not the only source. |
| Report update | High. Captures changed severity, location, notes, status, or metadata over time. | Medium because update paths can be sensitive to user latency and validation. | Medium because only explicit updates are captured; inferred transitions may be missed. | Recommended as one input with strict fail-open behavior. |
| Report clear | High. Captures resolution evidence and supports future duration analysis. | Medium because clear operations must never be blocked by capture. | Medium if clears can occur through multiple pathways. | Recommended as one input with lifecycle-normalized evidence. |
| Incident lifecycle transition | Very high. Captures normalized active, merged, resolved, expired, suppressed, or reopened transitions after production logic decides lifecycle state. | Lower if capture consumes a post-decision transition event; higher if it participates in lifecycle decisions. | Low to medium depending on whether all transition pathways emit consistently. | Preferred source for canonical passive evidence events. |
| Projection outputs | Medium to high. Captures derived historical interpretations from validated projection logic. | Low if captured as shadow evidence outside production; medium if treated as durable truth. | Medium because projections may represent derived snapshots rather than raw lifecycle facts. | Recommended only as secondary shadow evidence, not the primary capture trigger. |

### Preferred Source

The preferred source is a **post-decision incident lifecycle transition evidence event**, supplemented by report creation, report update, and report clear events as raw provenance inputs.

Rationale:

- Incident lifecycle transitions represent the point after production logic has already made its live decision, so capture can observe the outcome without influencing it.
- Lifecycle events can normalize multiple report-level operations into a consistent historical evidence contract.
- Raw report events preserve source provenance and help validate why a lifecycle transition occurred.
- Projection outputs remain valuable for validation and shadow analytics, but they should not be the primary persistence trigger because they are derived, not raw operational facts.

## 4. Recommended Capture Model

The recommended future model is an **append-only evidence archive fed by fail-open sidecar writes from post-decision lifecycle events and raw report provenance events**.

The model has four conceptual layers:

1. **Source observation layer:** observes completed production actions after the authoritative production action has succeeded.
2. **Evidence transformation layer:** converts the completed action into a historical evidence envelope with source, event type, entity identifiers, timestamps, payload digest, schema version, and capture metadata.
3. **Append-only persistence layer:** stores the evidence envelope as immutable historical evidence for later validation, analytics, retention review, and optional future projection comparison.
4. **Monitoring and validation layer:** observes capture health, write failures, duplicate rates, volume anomalies, and schema drift without affecting production behavior.

Recommended pattern:

- Use **sidecar writes** for raw report provenance evidence and lifecycle transition evidence.
- Treat projection-derived records as **shadow projection evidence** only when separately approved.
- Store evidence in an **append-only evidence archive** rather than mutable historical state.
- Avoid an event stream as the first production capture mechanism unless Gridly later needs cross-service fanout, replay, or high-volume queue semantics.

Why append-only:

- It preserves reversibility because evidence can be ignored or retention-managed without rewriting production state.
- It supports auditability because historical facts are not overwritten.
- It enables duplicate detection through idempotency keys and payload hashes.
- It avoids making historical tables authoritative for current incident state.

## 5. Write Path Architecture

The future write path should be designed as a best-effort sidecar sequence.

### Source

Candidate evidence should originate only after the production operation has completed its authoritative work. Sources may include:

- report created after successful report persistence;
- report updated after successful report persistence;
- report cleared after successful clear behavior;
- incident lifecycle transition after production lifecycle state has already been decided;
- projection output after shadow projection calculation, if separately approved.

### Transformation

The transformation layer should create a versioned evidence envelope. A future envelope could include:

- evidence schema version;
- capture source name;
- source operation type;
- source entity identifiers;
- incident identifiers when available;
- event timestamp and capture timestamp;
- lifecycle state before and after when available;
- normalized location reference when available;
- non-authoritative payload snapshot or digest;
- idempotency key;
- duplicate-detection hash;
- capture client or service version;
- fail-open error metadata for monitoring only.

Transformation must be pure and non-mutating. It must not update production report objects, incident objects, alert objects, awareness state, marker state, Route Watch state, or DriveTexas state.

### Persistence

Persistence should be append-only and isolated. A future implementation could use a dedicated historical evidence table or service-owned archive after a separate migration approval milestone. Persistence should support idempotency and duplicate detection, but duplicate handling must not block production completion.

Persistence must not be required for production correctness. The production operation is complete before evidence persistence is attempted.

### Failure Behavior

If transformation or persistence fails:

- log or count the capture failure for monitoring;
- do not throw into production flows;
- do not block the original operation;
- do not retry synchronously in a user-visible path;
- do not modify alert, awareness, marker, Route Watch, or incident lifecycle behavior;
- preserve enough diagnostic metadata to evaluate the capture system later.

## 6. Failure Isolation Design

The architecture isolates historical capture failures from protected systems by placing capture after production decisions and by making capture best-effort.

### Report Submission

Report submission must complete based on existing production validation and persistence only. Historical capture occurs only after submission success and cannot convert a successful report into a failed report.

### Alert Generation

Alerts must continue to use existing live production inputs. Historical capture must not be part of alert eligibility, alert suppression, alert routing, alert copy, or alert timing. If capture fails, alerts continue normally.

### Awareness

Awareness behavior must not read from or wait for historical evidence. Capture failures must not change awareness state, visibility, copy, prioritization, or timing.

### Markers

Marker rendering must remain based on existing live state. Historical evidence must not decide whether a marker exists, changes, clusters, styles, opens, closes, or refreshes.

### Route Watch

Route Watch must not depend on historical capture. Route Watch should not wait for capture, read evidence, change matching behavior, change relevance scoring, or alter route notifications due to capture success or failure.

### Incident Lifecycle

Incident lifecycle logic remains authoritative and runs before capture. Capture observes lifecycle transitions after they happen. It must not participate in merge, resolve, expire, suppress, reopen, or active-state decisions.

### Protected Systems

The future architecture must not modify or introduce dependencies into:

- `loadSharedReports()`;
- `activeHazards`;
- `getLiveHazardIncidents()`;
- `unifiedRoadIncidents`;
- `activeUnifiedIncidents`;
- alerts;
- awareness;
- markers;
- Route Watch;
- DriveTexas.

## 7. Disablement Architecture

Future capture must include immediate disablement controls before activation.

Recommended disablement layers:

- **Global capture flag:** a single kill switch that prevents all historical capture attempts.
- **Source-level flags:** separate controls for report creation, report update, report clear, lifecycle transition, and projection-output capture.
- **Persistence flag:** a control that allows transformation diagnostics without writes, if needed for safe investigation.
- **Environment gating:** explicit environment allowlist so non-approved environments cannot accidentally write evidence.
- **Operational runbook:** documented steps to disable capture, verify writes stopped, preserve diagnostics, and confirm production behavior remains unchanged.

Disablement must be immediate and fail-safe. If configuration cannot be loaded or is ambiguous, capture should default to disabled rather than risk unexpected writes.

## 8. Monitoring Architecture

Monitoring must observe capture health without driving production behavior.

### Write-Failure Monitoring

Track failed transformation and persistence attempts by source, event type, error class, and capture version. Failures should generate operational visibility, not user-visible errors.

### Volume Monitoring

Track evidence volume by source, event type, environment, and time bucket. Volume monitoring should detect sudden drops, spikes, or source imbalance that could indicate wiring problems or unexpected data storms.

### Duplicate Detection

Use idempotency keys and payload hashes to estimate duplicates. Duplicate detection should be monitored and reviewed, but duplicate write handling must remain isolated from production flow. Duplicate rates should be evaluated by source and event type.

### Health Validation

Define future validation checks such as:

- capture disabled state proof;
- capture enabled source inventory;
- expected event type coverage;
- write success rate;
- write latency distribution outside the user-visible path;
- schema-version distribution;
- malformed envelope rate;
- retention eligibility counts;
- sample traceability from raw report event to lifecycle evidence.

Monitoring output must not be consumed by alerts, awareness, markers, Route Watch, or live incident lifecycle logic.

## 9. Retention Architecture

Retention is a future governance concept only in V380. No retention jobs, SQL, migrations, deletion policies, or storage changes are approved.

Future retention design should address:

- **Retention classes:** separate raw report provenance, lifecycle transition evidence, projection-derived evidence, validation metadata, and failure diagnostics.
- **Retention periods:** define default retention windows before activation based on product, privacy, legal, analytics, and operational needs.
- **Deletion eligibility:** determine when evidence can be deleted, aggregated, anonymized, or compacted.
- **Hold policy:** define whether incidents under investigation, legal hold, or safety review require retention exceptions.
- **Audit trail:** record retention actions separately from production state.
- **Reversibility:** ensure retention cleanup does not affect current live reports, active incidents, alerts, awareness, markers, Route Watch, or DriveTexas behavior.

Retention ownership must exist before any production historical evidence writes are approved.

## 10. Ownership Model

### Evidence Ownership

Historical evidence should have a named engineering owner responsible for the evidence contract, source inventory, schema versioning, idempotency strategy, and capture boundary. Evidence ownership does not imply authority over live incident state.

### Validation Ownership

Validation should be owned by the historical program owner or a designated quality owner. This owner verifies source coverage, sample traceability, duplicate rates, malformed payload rates, and alignment with the approved evidence contract.

### Monitoring Ownership

Monitoring should be owned by operations or the service owner responsible for production observability. Monitoring ownership includes dashboards, thresholds, failure triage, and disablement verification.

### Retention Ownership

Retention should be jointly owned by engineering, product, privacy/legal, and operations before activation. No production retention behavior should exist without a named owner and documented policy.

## 11. Architecture Recommendation

Gridly should adopt, in a future separately approved milestone, an **append-only passive evidence archive** populated by **fail-open sidecar capture** from **post-decision incident lifecycle transitions**, supplemented by raw report creation, update, and clear provenance events.

This architecture is preferred because it best satisfies the V380 principle:

> Capture Everything. Show Nothing. Depend On Nothing.

Reasons:

- **Passive:** capture observes completed production actions rather than initiating or controlling them.
- **Non-authoritative:** evidence is stored for future analysis, not for live decision-making.
- **Fail-open:** failures stay inside the capture boundary and cannot block production paths.
- **Reversible:** capture can be disabled, ignored, or removed without breaking production behavior.
- **Invisible:** no history UI, historical reads, map changes, marker changes, alerts changes, awareness changes, or Route Watch changes are required.
- **Auditable:** append-only evidence enables later validation of source coverage, lifecycle timing, duplicates, and retention.
- **Compatible with prior milestones:** it preserves the projection-first and schema-governed posture established by V363 through V379 without executing migrations or changing Supabase.

An event stream may become appropriate later if Gridly needs replay, backpressure management, cross-service fanout, or asynchronous worker isolation at higher scale. V380 does not require that complexity as the initial architecture recommendation.
