## 2. Passive Capture Definition

For Gridly, passive historical capture means a future, separately approved sidecar mechanism that preserves evidence generated by already-approved live workflows without changing those workflows.

A capture mechanism qualifies as passive only if all of the following are true:

- It is **write-only from the perspective of history**: it may preserve evidence, but no production system reads history for live decisions.
- It is **non-authoritative**: historical records cannot determine active incident state, alert eligibility, awareness wording, marker rendering, Route Watch relevance, DriveTexas behavior, or report lifecycle outcomes.
- It is **fail-open**: if capture fails, times out, is disabled, is rate-limited, or loses credentials, live Gridly behavior continues successfully.
- It is **sidecar-only**: capture occurs adjacent to existing production flows and does not become part of the required success path for report creation, report updates, incident closure, or lifecycle transitions.
- It is **invisible to users**: no history UI, labels, badges, alerts, map layers, filters, popups, panels, notifications, or copy changes are introduced.
- It is **additive-only**: future storage must not rewrite, delete, repurpose, or backfill existing production data unless a separate milestone approves that work.
- It is **disable-safe**: operations can turn capture off without requiring application rollback, data restoration, production schema rollback, or user-facing degradation.
- It is **auditable**: capture attempts, failures, disablement state, and volume can be reviewed without exposing sensitive data unnecessarily.

The following do **not** qualify as passive capture:

- Any historical read by alerts, awareness, markers, Route Watch, lifecycle logic, or UI.
- Any synchronous requirement that a historical write succeed before a user report or lifecycle transition succeeds.
- Any trigger or function on production tables that can reject, delay, mutate, or alter production writes.
- Any capture path that changes `loadSharedReports()`, `activeHazards`, `getLiveHazardIncidents()`, `unifiedRoadIncidents`, `activeUnifiedIncidents`, alerts, awareness, markers, Route Watch, or DriveTexas.
- Any user-visible history feature.

## 3. Candidate Capture Sources

V379 does not implement capture. It only evaluates possible future capture points.

| Candidate source | Potential evidence value | Passive suitability | Required safeguards before implementation |
| --- | --- | --- | --- |
| Report creation | Captures initial user evidence, category, location, timestamps, and source metadata at the point evidence first appears. | Potentially suitable if capture is asynchronous and not required for report creation success. | Report submission must complete even if capture fails; no production insert trigger may block reports; captured fields must be minimized and governed. |
| Report update | Captures changing status, corrected details, lifecycle evolution, or moderation/normalization outcomes. | Potentially suitable only if update capture is append-only and cannot mutate the live report. | Update flow must not wait on historical writes; failures must be logged outside user-facing flows; duplicate/version handling must be defined. |
| Incident closure | Captures `closed_at`, duration, final status, and resolution evidence. | High value, but sensitive because closure is a lifecycle boundary. | Closure must not depend on historical storage; capture cannot prevent closure, reopen incidents, alter closure timing, or change active-state calculations. |
| Lifecycle transitions | Captures active → stale → closed or similar transitions for later duration and recurrence analysis. | Suitable only as a sidecar observer. | Transition logic remains authoritative in live systems; capture cannot influence state machine decisions; event ordering and idempotency must be defined. |
| Incident aggregation/projection output | Captures summarized incident shape after live reports are grouped. | Strategically useful for future recurrence/hotspot analysis. | Must not change aggregation logic or become a source for live aggregation; summary capture must be detached from rendering and alert decisions. |
| Periodic snapshot job | Captures state at intervals independent of user actions. | Potentially suitable but adds scheduler and volume complexity. | Scheduler must be independently disableable; missed snapshots cannot affect live operation; volume caps and dedupe rules are required. |

Initial planning preference: if Gridly later proceeds, the safest candidate is a narrowly scoped append-only sidecar event capture around already-successful lifecycle outcomes, not a production-table trigger that can block live writes.

## 4. Fail-Open Requirements

A future passive capture design must prove that historical capture failure cannot affect the following systems:

- **Reporting**
  - Report creation and update success must not depend on historical capture success.
  - User feedback must not mention capture failure.
  - Capture latency must not delay the report submission completion path.
- **Alerts**
  - Alert eligibility, timing, severity, suppression, copy, deduplication, and notification behavior must not read or depend on historical storage.
  - Capture failure must not suppress or create alerts.
- **Awareness**
  - Awareness text, ranking, severity, confidence, nearby context, and recurrence language must not read or depend on historical storage.
  - Capture failure must not alter awareness content.
- **Markers**
  - Marker creation, icons, ordering, clustering, lifecycle, and visibility must continue to use only approved live sources.
  - Capture failure must not remove, add, delay, or restyle markers.
- **Route Watch**
  - Route relevance, route risk, geometry scoring, monitoring activation, and commute intelligence must not read or depend on historical storage.
  - Capture failure must not affect Route Watch state or output.
- **Incident lifecycle**
  - Active, stale, cleared, merged, or closed states must remain governed by live production rules.
  - Capture failure must not prevent closure, reactivate closed incidents, change merge behavior, or rewrite live records.

Minimum proof expected in a future implementation plan:

- Explicit asynchronous or post-success sequencing.
- Timeouts and error swallowing around capture side effects.
- Idempotency keys or dedupe strategy that prevents retries from corrupting historical evidence.
- Tests or traces showing live success when capture is disabled, unavailable, slow, or returning errors.
- No production logic branch that uses historical write success as a condition.

## 5. Isolation Requirements

Future passive capture planning must define both technical and operational isolation.

Technical isolation expectations:

- Historical storage remains separate from live production tables and does not alter `public.reports` or any existing production table.
- No historical read path is introduced into `loadSharedReports()`, `activeHazards`, `getLiveHazardIncidents()`, `unifiedRoadIncidents`, `activeUnifiedIncidents`, alerts, awareness, markers, Route Watch, or DriveTexas.
- Any writer uses least-privilege credentials appropriate to the approved design and does not expose service-role secrets to clients.
- Capture code is separated behind a clearly named module, adapter, queue, job, or sidecar boundary.
- Capture payloads are immutable evidence events or summaries, not production commands.
- Capture operations are bounded by strict timeout, retry, and backoff behavior.
- Historical tables cannot become foreign-key prerequisites for live production writes.
- No database trigger may be added to a production table unless a later milestone proves it cannot block, mutate, or reject production behavior; the preferred posture is no production-table triggers.

Operational isolation expectations:

- Capture has an owner, disablement procedure, monitoring expectations, and incident-response path.
- Capture deployment is separate from user-facing historical feature activation.
- Capture rollout can be stopped independently from the application.
- Capture evidence is reviewed under privacy and governance controls.
- Any future migration, writer deployment, scheduler, queue, edge function, or job runner receives separate approval.

## 6. Disablement Requirements

A future passive capture implementation must be disableable without affecting Gridly operation.

Minimum disablement requirements:

- A documented kill switch or configuration flag can stop capture attempts.
- Disablement does not require changing alert, awareness, marker, Route Watch, lifecycle, or report code paths.
- Disablement does not require dropping historical tables.
- Disablement does not require deleting captured records.
- Disablement does not require rolling back production application behavior.
- Disablement can be verified by logs or metrics showing capture attempts stopped.
- Re-enablement, if allowed, must be explicit and auditable.
- If credentials are removed or unavailable, live production remains healthy.

Acceptable future disablement models may include a server-side feature flag, disabled queue consumer, disabled scheduled job, revoked writer credential, or deployment-level environment variable. The chosen model must be documented before implementation.

## 7. Monitoring Requirements

Minimum monitoring and audit expectations for future passive capture:

- Capture attempt count.
- Capture success count.
- Capture failure count and failure class.
- Capture latency distribution or timeout count.
- Queue depth or backlog, if a queue is used.
- Retry count and dead-letter count, if retries are used.
- Duplicate/idempotency rejection count.
- Daily captured event volume by event type.
- Storage growth estimate.
- Disablement state and last state change.
- Writer identity or job identity, without exposing secrets.
- Redacted sample payload audit confirming only approved fields are stored.

Monitoring must not create user-visible alerts unless a separate operational alerting policy is approved. Capture monitoring should notify maintainers, not end users, and must never change production incident behavior.

## 8. Retention / Volume Review

Historical evidence accumulation creates storage, index, and lifecycle obligations even if no user-facing history exists.

Planning considerations:

- Define expected event types and approximate daily event counts before execution.
- Estimate storage growth for raw events, incident summaries, indexes, and recurrence records.
- Prefer append-only evidence with dedupe/idempotency over repeated full snapshots unless a snapshot use case is approved.
- Avoid storing unbounded payloads or full client/browser state.
- Define whether raw event evidence and derived summaries have different retention periods.
- Define whether pre-beta evidence receives special handling if test or seed data is present.
- Define cleanup procedures before volume becomes operationally significant.
- Define whether retention is time-based, volume-based, milestone-based, or manual-review-based.
- Confirm rollback and disablement plans do not accidentally destroy evidence that governance expects to preserve.

V379 does not choose a retention period. It requires a future passive-capture implementation milestone to choose and justify one before any writes begin.

## 9. Privacy / Governance Review

Passive evidence accumulation may be invisible to users, but it is still data collection and must be governed accordingly.

Governance concerns:

- Field minimization: capture only fields necessary for future recurrence, duration, hotspot, lifecycle, or quality analysis.
- Provenance: record source and model/version context so future consumers know evidence origin and limitations.
- User data: avoid unnecessary personal identifiers, raw device identifiers, precise user metadata, or free-form content unless specifically approved.
- Location sensitivity: incident location evidence is core to Gridly but still requires retention and access discipline.
- Test data: pre-beta test, fake, or internal records must be identifiable or excluded before being treated as real historical evidence.
- Access control: historical records must remain unavailable to anonymous and authenticated client reads unless a later milestone approves reads.
- Service credentials: any writer credential must be protected and never shipped to client code.
- Deletion/correction: future governance must define whether and how historical evidence can be corrected, redacted, or deleted.
- Product boundary: preserved evidence must not be presented as historical truth to users until separate validation, UI, and activation milestones approve that use.
