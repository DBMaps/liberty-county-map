## 2. Architecture Summary

The approved V380 architecture is an append-only passive evidence archive. It is intended to preserve evidence generated by completed production workflows without making historical storage authoritative for any live behavior.

Core posture:

- **Capture Everything:** future capture should preserve raw report creation, report update, report clear, and post-decision incident lifecycle transition evidence when separately approved.
- **Show Nothing:** no history UI, labels, layers, filters, popups, alerts, awareness copy, marker changes, Route Watch surfaces, or public history features are introduced.
- **Depend On Nothing:** live Gridly behavior must not require historical tables, historical writes, capture queues, evidence readers, analytics jobs, or successful archive persistence.

Recommended future model:

1. **Source observation layer** observes already-completed production actions.
2. **Evidence transformation layer** converts those actions into versioned, immutable evidence envelopes.
3. **Append-only persistence layer** stores non-authoritative evidence for later validation, audit, analytics, and retention review.
4. **Monitoring and validation layer** observes health, latency, failures, duplicates, volume, schema version drift, and disablement state without driving production behavior.

The preferred capture source is a post-decision incident lifecycle transition event, supplemented by raw report creation, report update, and report clear provenance events. Lifecycle logic remains authoritative and runs before capture. Capture observes lifecycle decisions after they occur; it does not participate in merge, resolve, expire, suppress, reopen, active-state, alert, awareness, marker, Route Watch, or DriveTexas decisions.

Protected production systems remain unchanged and must not depend on historical capture:

- `loadSharedReports()`
- `activeHazards`
- `getLiveHazardIncidents()`
- `unifiedRoadIncidents`
- `activeUnifiedIncidents`
- alerts

## 3. Future Feature Flag Strategy

V381 does not implement feature flags. A later implementation-design milestone should define feature-flag and configuration concepts before any migration execution or write activation.

Future concepts should include:

- `GRIDLY_ENABLE_PASSIVE_HISTORY_CAPTURE`
  - Global kill switch for all passive historical capture attempts.
  - Default must be disabled in ambiguous, missing, invalid, or unapproved configurations.
- Source-level gates:
  - `GRIDLY_ENABLE_HISTORY_CAPTURE_REPORT_CREATED`
  - `GRIDLY_ENABLE_HISTORY_CAPTURE_REPORT_UPDATED`
  - `GRIDLY_ENABLE_HISTORY_CAPTURE_REPORT_CLEARED`
  - `GRIDLY_ENABLE_HISTORY_CAPTURE_LIFECYCLE_TRANSITION`
  - Optional future `GRIDLY_ENABLE_HISTORY_CAPTURE_SHADOW_PROJECTION`, only if separately approved.
- Write gate:
  - A separate persistence/write gate that can allow envelope construction diagnostics while preventing archive writes during validation.
- Environment allowlist:
  - Capture writes should only be possible in explicitly approved environments.
  - Unknown environments should behave as disabled.
- Emergency disablement controls:
  - Operators must be able to stop all capture attempts immediately without rolling back the application, dropping tables, deleting evidence, or modifying protected production systems.
- Credential disablement:
  - Removing or revoking writer credentials must not affect report creation, report update, report clear, lifecycle transitions, alerts, awareness, markers, Route Watch, or DriveTexas behavior.
- Auditability:
  - Flag state, source-level enablement, write-gate state, environment identity, and last disablement action should be observable by maintainers.

No flag names are approved as implementation API in V381. They are planning labels for a future design milestone.

## 4. Future Capture Flow

V381 does not implement capture. A future implementation-design milestone should document and later test the exact passive archival flow.

### Report creation

1. Existing report creation validation and persistence complete using current production behavior.
2. User-facing report creation completes independent of history.
3. A best-effort sidecar observes the completed creation result.
4. A versioned evidence envelope is built with source operation, source identifiers, event timestamp, capture timestamp, payload digest, idempotency key, schema version, and approved provenance fields.
5. The envelope is appended to historical evidence storage only if capture and write gates allow it.
6. Any capture failure is logged or counted for monitoring only and does not affect the created report.

### Report update

1. Existing report update behavior completes under current production rules.
2. Live state, alerts, awareness, markers, Route Watch, and lifecycle logic remain governed by current approved inputs.
3. A best-effort sidecar observes the completed update.
4. The evidence envelope records approved before/after or changed-field provenance where available, without mutating the live report.
5. Persistence is append-only and non-authoritative.
6. Failures remain isolated and invisible to users.

### Report clear

1. Existing clear behavior completes under current production rules.
2. The clear action remains authoritative only in the live system.
3. A best-effort sidecar observes the completed clear result.
4. The evidence envelope records clear provenance, final report status, timestamps, source identifiers, and approved lifecycle context where available.
5. Archive writes are attempted only after the clear action has already succeeded.
6. Capture failure cannot prevent clearing, reopen an incident, alter active-state calculations, or change alerts, awareness, markers, or Route Watch.

### Incident lifecycle transition

1. Existing incident lifecycle logic decides the transition first.
2. Production state proceeds according to existing behavior.
3. A sidecar observes the completed post-decision transition.
4. The evidence envelope records event type, prior and next lifecycle states where available, transition timestamp, incident identifiers where available, source report references where approved, payload digest, idempotency key, and schema version.
5. Append-only storage preserves the transition as non-authoritative evidence.
6. Capture never participates in deciding active, merged, resolved, expired, suppressed, reopened, or cleared states.

## 5. Future Failure Handling Plan

Passive capture must remain fail-open. A future implementation must prove the production action completes successfully even when capture is disabled, unavailable, slow, duplicated, malformed, rate-limited, or failing.

Expected future behavior:

- **Write failures**
  - Do not throw into production flows.
  - Do not reject reports, updates, clears, or lifecycle transitions.
  - Record failure class, source, event type, capture version, and redacted diagnostics for maintainers.
  - Avoid synchronous retries that increase user-visible latency.
- **Storage failures**
  - Treat archive storage as optional from the perspective of production.
  - Continue live behavior if the historical table, archive endpoint, queue, job, or credential is unavailable.
  - Escalate through operational monitoring only.
- **Timeout conditions**
  - Use strict capture-side timeouts.
  - On timeout, abandon or enqueue according to the later approved design without blocking the originating production operation.
  - Track timeout counts separately from validation or schema failures.
- **Duplicate events**
  - Use idempotency keys and payload hashes to detect duplicate capture attempts.
  - Duplicate detection may suppress duplicate archive writes or record duplicate metrics, but it must never affect live production behavior.
  - Duplicate rates should be monitored by source and event type.
- **Malformed envelopes**
  - Reject from historical persistence or quarantine only within the capture boundary.
  - Do not mutate or correct live production data as part of capture repair.
- **Configuration ambiguity**
  - Default capture to disabled if flag state, environment allowlist, credential state, or write-gate state is unclear.

## 6. Future Validation Plan

Before implementation, Gridly should require validation plans and acceptance criteria for the passive capture path.

Required validation categories:

- **Parity validation**
  - Verify protected production systems produce identical user-visible behavior with capture disabled and enabled.
  - Confirm no changes to reporting, alerts, awareness, markers, Route Watch, DriveTexas, or live lifecycle outcomes.
- **Production isolation validation**
  - Verify no historical reads are introduced into `loadSharedReports()`, `activeHazards`, `getLiveHazardIncidents()`, `unifiedRoadIncidents`, `activeUnifiedIncidents`, alerts, awareness, markers, Route Watch, or DriveTexas.
  - Verify no production logic branches on historical write success.
- **Duplicate detection validation**
  - Verify idempotency keys and payload hashes identify repeated attempts without corrupting archived evidence or influencing live behavior.
- **Failure-path validation**
  - Simulate archive write failure, storage unavailability, credential revocation, timeout, malformed payload, duplicate event, and disabled flag states.
  - Confirm the originating report or lifecycle operation still succeeds.
- **Latency validation**
  - Confirm capture does not add user-visible latency to report creation, update, clear, alerts, awareness, markers, Route Watch, or lifecycle transitions.
- **Volume validation**
  - Estimate event volume by source and confirm storage growth, index cost, monitoring capacity, and retention assumptions.
- **Privacy and field validation**
  - Confirm only approved fields are captured.
  - Confirm free-form content, personal identifiers, device metadata, and precise provenance are minimized or governed.
- **Environment validation**
  - Confirm capture cannot write from unapproved environments.
- **Rollback validation**
  - Confirm capture can be disabled without application rollback, schema rollback, data deletion, or production behavior change.

## 7. Future Monitoring Plan

A future implementation-design milestone should define monitoring before writes begin.

Minimum monitoring and audit requirements:

- Capture enabled/disabled state by environment.
- Source-level flag state.
- Write-gate state.
- Capture attempt count by source and event type.
- Capture success count by source and event type.
- Capture failure count by failure class, source, event type, and capture version.
- Timeout count and latency distribution outside the user-visible path.
- Duplicate/idempotency detection count.
- Malformed envelope count.
- Archive write latency and storage error rate.
- Daily event volume and volume anomaly detection by source.
- Storage growth estimates.
- Schema-version distribution.
- Redacted sample-payload audit results.
- Writer identity, job identity, or service identity without exposing secrets.
- Last disablement or re-enablement action and operator/runbook reference.

Monitoring must be operational only. It must not create user-visible history, alter alerts, alter awareness, alter markers, alter Route Watch, alter DriveTexas behavior, or become an input to live incident lifecycle logic.

## 8. Future Rollback / Disablement Plan

Disablement must be designed before capture is implemented.

Future disablement requirements:

- A global kill switch can stop all capture attempts immediately.
- Source-level controls can disable specific capture sources independently.
- A write gate can stop persistence while allowing approved diagnostics, if needed.
- Capture defaults to disabled when configuration is missing, ambiguous, invalid, or from an unapproved environment.
- Revoking writer credentials does not affect production behavior.
- Disabling queue consumers, scheduled jobs, edge functions, or sidecar workers does not affect production behavior.
- Disablement does not require dropping historical tables.
- Disablement does not require deleting archived evidence.
- Disablement does not require application rollback.
- Disablement does not require changes to alerts, awareness, markers, Route Watch, DriveTexas, `loadSharedReports()`, `activeHazards`, `getLiveHazardIncidents()`, `unifiedRoadIncidents`, or `activeUnifiedIncidents`.
- Operators can verify writes stopped through logs or metrics.
- Re-enablement is explicit, auditable, and separately reviewed when needed.

Rollback planning for migration execution remains out of scope for V381 and must be revisited before any future migration execution milestone.
